# DQL：基础查询、条件查询与模糊查询

## 本章导读

前面几章我们学的都是“怎么把数据放进数据库”（DDL 建表、DML 增删改）。从本章开始，我们进入 MySQL 里**使用频率最高、最重要**的部分——**DQL（Data Query Language，数据查询语言）**，也就是 `SELECT` 查询。

打个比方：如果说数据库是一个巨大的仓库，前面几章是教你“怎么把货物搬进仓库、怎么摆放、怎么换货”，那么本章开始就是教你**“怎么从仓库里精准地找到你想要的东西”**。在真实工作中，你 90% 以上的时间都在写查询语句——报表、列表、统计、接口数据……几乎都是 `SELECT`。所以这一块务必学扎实。

`SELECT` 是一个“大块头”，它的完整语法有 7 个子句。本章我们先把**全貌**给你看清楚，然后聚焦讲透前三个最核心的部分：

- **SELECT**（查哪些列）
- **FROM**（从哪张表查）
- **WHERE**（满足什么条件才查出来）

至于 `GROUP BY`（分组）、`HAVING`（分组后过滤）、`ORDER BY`（排序）、`LIMIT`（分页），会在后续章节继续深入。本章是 DQL 的“地基”，地基打牢了，后面的高楼才稳。

> 本章沿用全套教程统一的示例数据库 `db_learn`，主要用到部门表 `dept` 和员工表 `emp`。

## 1. DQL 的地位与 SELECT 完整语法骨架

### 1.1 为什么 DQL 最重要

数据库的核心价值是**“存”和“取”**。存进去的数据如果取不出来、取不准，就毫无意义。而 DQL 就是负责“取”的那把钥匙。它的特点是：

- **不修改数据**：`SELECT` 只读，再怎么查也不会改坏表里的数据，所以可以放心大胆地练。
- **使用最频繁**：日常开发中绝大多数 SQL 都是查询。
- **变化最丰富**：从简单的“查一列”到复杂的多表关联、子查询、聚合统计，全靠它。

### 1.2 SELECT 的完整语法骨架（先看全貌）

先把这个“骨架”刻在脑子里，后面所有查询都是它的子集：

```sql
SELECT   字段列表           -- ① 要查哪些列（最后才执行，但写在最前面）
FROM     表名               -- ② 从哪张表查
WHERE    条件               -- ③ 行级过滤（分组前的条件）
GROUP BY 分组字段           -- ④ 按什么分组
HAVING   分组后条件         -- ⑤ 组级过滤（分组后的条件）
ORDER BY 排序字段 [ASC|DESC]-- ⑥ 按什么排序
LIMIT    起始, 条数;        -- ⑦ 分页/限制行数
```

逐项速览（本章只深入①②③）：

| 子句       | 作用             | 通俗理解                     | 本章是否深入 |
|------------|------------------|------------------------------|--------------|
| `SELECT`   | 选择要显示的列   | “我想看哪几列”               | ✅ 是        |
| `FROM`     | 指定数据来源表   | “去哪张表里翻”               | ✅ 是        |
| `WHERE`    | 行过滤条件       | “只要满足条件的那些行”       | ✅ 是        |
| `GROUP BY` | 分组             | “把同类的归成一堆”           | 后续章节     |
| `HAVING`   | 分组后过滤       | “只要满足条件的那些组”       | 后续章节     |
| `ORDER BY` | 排序             | “按某列从大到小/小到大排”    | 后续章节     |
| `LIMIT`    | 限制行数/分页    | “只看前 10 条”               | 后续章节     |

> ⚠️ 注意：上面这个**书写顺序是固定的**，不能随意调换。比如不能把 `WHERE` 写到 `FROM` 前面，否则报语法错误。记忆口诀：**“selete（选） from（从） where（哪） group（组） having（再筛） order（排） limit（截）”**。

> 💡 提示：书写顺序 ≠ 执行顺序。MySQL 实际执行时大致是 `FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT`，这也是为什么 `SELECT` 里起的别名常常不能在 `WHERE` 里用（`WHERE` 比 `SELECT` 先执行）。这个细节本章末尾的“常见坑”会再提。

## 2. 基础查询（SELECT ... FROM ...）

最简单的查询只需要 `SELECT` + `FROM` 两部分：选哪些列、从哪张表。

### 2.1 查询全部列：SELECT *

**语法格式：**

```sql
SELECT * FROM 表名;
```

- `*` 是通配符，代表“这张表的**所有列**”。

**示例：查询 emp 表的全部员工信息**

```sql
SELECT * FROM emp;
```

**执行结果：**

| id | ename | gender | salary | join_date  | dept_id | bonus |
|----|-------|--------|--------|------------|---------|-------|
| 1  | 张三  | 男     | 8000   | 2020-01-10 | 1       | 1000  |
| 2  | 李四  | 男     | 12000  | 2019-03-15 | 1       | NULL  |
| 3  | 王五  | 女     | 9500   | 2021-06-01 | 2       | 2000  |
| 4  | 赵六  | 女     | 6000   | 2022-09-20 | 2       | NULL  |
| 5  | 孙七  | 男     | 15000  | 2018-11-05 | 3       | 3000  |

> 🕳️ 常见坑：`SELECT *` 在**学习、临时调试**时很方便，但在**正式代码（尤其是 Java / 后端接口）里强烈不建议用**！原因有三：
> 1. **性能差**：会把不需要的大字段（如长文本、blob）也查出来，浪费网络和内存。
> 2. **不稳定**：哪天表加了一列，`*` 的返回结构就变了，可能把上层代码搞崩。
> 3. **可读性差**：别人看代码不知道你到底用了哪些列。
>
> 正确做法是**明确写出要查的列**，下面就讲。

### 2.2 查询指定列

**语法格式：**

```sql
SELECT 列名1, 列名2, ... FROM 表名;
```

- 列之间用逗号 `,` 隔开。
- 显示顺序就是你写的顺序（可以和表里定义的顺序不同）。

**示例：只查员工的姓名、工资、所属部门编号**

```sql
SELECT ename, salary, dept_id FROM emp;
```

**执行结果：**

| ename | salary | dept_id |
|-------|--------|---------|
| 张三  | 8000   | 1       |
| 李四  | 12000  | 1       |
| 王五  | 9500   | 2       |
| 赵六  | 6000   | 2       |
| 孙七  | 15000  | 3       |

**示例：调整列的显示顺序（先工资后姓名）**

```sql
SELECT salary, ename FROM emp;
```

**执行结果：**

| salary | ename |
|--------|-------|
| 8000   | 张三  |
| 12000  | 李四  |
| 9500   | 王五  |
| 6000   | 赵六  |
| 15000  | 孙七  |

可见，**`SELECT` 后面列的先后顺序，决定了结果中列的先后顺序**。

### 2.3 列的别名 AS（可省略 AS）

查询出来的列名默认就是字段名（如 `ename`、`salary`），但有时我们希望显示得更友好（比如显示中文“姓名”“月薪”），这时用**别名（alias）**。

**语法格式：**

```sql
SELECT 列名 AS 别名, 列名 别名 FROM 表名;
```

- `AS` 关键字用来给列起别名。
- **`AS` 可以省略**，直接在列名后面空一格写别名即可。
- 如果别名里**包含空格或特殊字符（如中文标点）**，建议用**单引号或反引号**括起来更稳妥。

**示例 1：用 AS 起中文别名**

```sql
SELECT ename AS 姓名, salary AS 月薪 FROM emp;
```

**执行结果：**

| 姓名 | 月薪  |
|------|-------|
| 张三 | 8000  |
| 李四 | 12000 |
| 王五 | 9500  |
| 赵六 | 6000  |
| 孙七 | 15000 |

**示例 2：省略 AS（效果完全一样）**

```sql
SELECT ename 姓名, salary 月薪 FROM emp;
```

结果与示例 1 **完全相同**。`AS` 写不写都行，写上更清晰、不容易让人误解。

**示例 3：别名带空格时用引号括起来**

```sql
SELECT ename AS '员工 姓名', salary AS '月 薪' FROM emp;
```

**执行结果（列标题里就能带空格了）：**

| 员工 姓名 | 月 薪 |
|-----------|-------|
| 张三      | 8000  |
| ...       | ...   |

> 💡 提示：别名主要是**给查询结果“改个显示名”**，**不会真的改表结构**。表里字段名还是 `ename`、`salary`，别名只在这一次查询结果中生效。

### 2.4 表别名

不仅列可以起别名，**表**也可以。表别名在**多表查询**时特别有用（可以少写很多字），单表查询里也能用。

**语法格式：**

```sql
SELECT 表别名.列名 FROM 表名 AS 表别名;   -- AS 同样可省略
```

**示例：给 emp 表起别名 e**

```sql
SELECT e.ename, e.salary FROM emp AS e;
-- 等价写法（省略 AS）：
-- SELECT e.ename, e.salary FROM emp e;
```

**执行结果：**

| ename | salary |
|-------|--------|
| 张三  | 8000   |
| 李四  | 12000  |
| 王五  | 9500   |
| 赵六  | 6000   |
| 孙七  | 15000  |

> ⚠️ 注意：**一旦给表起了别名，原来的表名在这条语句里就不能再用了**。比如上面起了别名 `e`，再写 `emp.ename` 就会报错，必须写 `e.ename`。
>
> 表别名的真正威力会在后续“多表连接查询”章节体现，例如 `SELECT e.ename, d.dept_name FROM emp e, dept d`，用一个字母代替长表名，语句清爽很多。

## 3. 去重、计算列与 NULL 处理

### 3.1 去重 DISTINCT

有时一列里有很多重复值，我们只想看“有哪些不同的值”，这时用 `DISTINCT`（去重）。

**语法格式：**

```sql
SELECT DISTINCT 列名 FROM 表名;
```

- `DISTINCT` 紧跟在 `SELECT` 后面。
- 它对**后面所有列的组合**去重，而不是只对第一列去重。

**示例 1：员工分布在哪些部门（去掉重复的 dept_id）**

不去重的话：

```sql
SELECT dept_id FROM emp;
```

| dept_id |
|---------|
| 1       |
| 1       |
| 2       |
| 2       |
| 3       |

1 和 2 各出现了两次。加上 `DISTINCT`：

```sql
SELECT DISTINCT dept_id FROM emp;
```

**执行结果（重复的被合并）：**

| dept_id |
|---------|
| 1       |
| 2       |
| 3       |

**示例 2：DISTINCT 作用于多列时，是“整行组合”去重**

```sql
SELECT DISTINCT gender, dept_id FROM emp;
```

会把 `(gender, dept_id)` **这个组合**作为一个整体去重：

| gender | dept_id |
|--------|---------|
| 男     | 1       |
| 女     | 2       |
| 男     | 3       |

> 🕳️ 常见坑：很多初学者以为 `SELECT DISTINCT gender, dept_id` 是“只对 gender 去重”，其实是**对 gender 和 dept_id 的组合**去重。只要两列的值不是完全相同，就会被保留。比如上面 `(男,1)` 和 `(男,3)` 因为 dept_id 不同，都被保留了下来。

### 3.2 查询时进行计算（数值列运算）

`SELECT` 后面不仅能写列名，还能写**表达式**，对数值列做加减乘除运算，结果作为新的一列显示出来。

**支持的算术运算符：**

| 运算符 | 含义     | 示例           |
|--------|----------|----------------|
| `+`    | 加       | `salary + 100` |
| `-`    | 减       | `salary - 100` |
| `*`    | 乘       | `salary * 12`  |
| `/`    | 除       | `salary / 2`   |
| `%`    | 取余/取模 | `salary % 1000`|

**示例 1：计算每位员工的“年薪”（月薪 × 12）**

```sql
SELECT ename, salary, salary * 12 AS 年薪 FROM emp;
```

**执行结果：**

| ename | salary | 年薪   |
|-------|--------|--------|
| 张三  | 8000   | 96000  |
| 李四  | 12000  | 144000 |
| 王五  | 9500   | 114000 |
| 赵六  | 6000   | 72000  |
| 孙七  | 15000  | 180000 |

> 💡 提示：计算列**一定要起别名**！如果不起别名，列标题就会原样显示成 `salary * 12`，又长又难看，前端/Java 拿到这个奇怪的列名也不好处理。

**示例 2：涨薪 500 后的工资**

```sql
SELECT ename, salary AS 原工资, salary + 500 AS 涨薪后 FROM emp;
```

**执行结果：**

| ename | 原工资 | 涨薪后 |
|-------|--------|--------|
| 张三  | 8000   | 8500   |
| 李四  | 12000  | 12500  |
| 王五  | 9500   | 10000  |
| 赵六  | 6000   | 6500   |
| 孙七  | 15000  | 15500  |

> ⚠️ 注意：这种计算**只影响查询结果的“显示”，不会真的修改表里的数据**。`emp` 表里 `salary` 仍然是原值。要真正改数据得用 `UPDATE`（DML），那是另一回事。

### 3.3 NULL 的处理：IFNULL

现在我们来对付那个埋好的“坑”——`bonus` 列里的 `NULL`。

先理解 `NULL` 是什么：**`NULL` 表示“未知 / 没有值 / 空”，它不是数字 0，也不是空字符串 `''`**，而是“什么都没有”。

这会带来一个很坑的问题：**任何数值和 `NULL` 做运算，结果都是 `NULL`！**

**反面示例：直接计算“工资 + 奖金”**

```sql
SELECT ename, salary, bonus, salary + bonus AS 月收入 FROM emp;
```

**执行结果（注意李四、赵六的“月收入”变成了 NULL）：**

| ename | salary | bonus | 月收入 |
|-------|--------|-------|--------|
| 张三  | 8000   | 1000  | 9000   |
| 李四  | 12000  | NULL  | **NULL** |
| 王五  | 9500   | 2000  | 11500  |
| 赵六  | 6000   | NULL  | **NULL** |
| 孙七  | 15000  | 3000  | 18000  |

李四明明有 12000 的工资，结果月收入却显示 `NULL`！这显然不符合我们的预期。原因就是 `12000 + NULL = NULL`。

**解决办法：用 `IFNULL` 函数把 `NULL` 替换成默认值（比如 0）。**

**语法格式：**

```sql
IFNULL(列名, 默认值)
```

- 如果“列名”的值**不是 NULL**，就返回它本身；
- 如果“列名”的值**是 NULL**，就返回“默认值”。

**正确示例：把没奖金的当成 0 来算**

```sql
SELECT ename,
       salary,
       bonus,
       salary + IFNULL(bonus, 0) AS 月收入
FROM emp;
```

**执行结果（这下对了）：**

| ename | salary | bonus | 月收入 |
|-------|--------|-------|--------|
| 张三  | 8000   | 1000  | 9000   |
| 李四  | 12000  | NULL  | 12000  |
| 王五  | 9500   | 2000  | 11500  |
| 赵六  | 6000   | NULL  | 6000   |
| 孙七  | 15000  | 3000  | 18000  |

`IFNULL(bonus, 0)` 把李四、赵六的 `NULL` 当成 0，于是 `12000 + 0 = 12000`，结果正常了。

> 💡 提示：`IFNULL` 在做**计算和聚合统计**（如求工资奖金总和、平均收入）时几乎是必备的，否则一个 `NULL` 就能把整列结果污染掉。后续讲聚合函数（`SUM`、`AVG`）时还会用到它。

> 🕳️ 常见坑：`IFNULL` 是 **MySQL 特有**的函数。标准 SQL 里类似功能叫 `COALESCE(列, 默认值)`（`COALESCE` 还能写多个参数，返回第一个非 NULL 的值），跨数据库时更通用。本套教程以 MySQL 为准，用 `IFNULL` 即可。

## 4. 条件查询 WHERE

前面查的都是“整张表所有行”。但真实场景里我们往往只要**满足某些条件的行**——比如“工资大于 1 万的员工”“研发部的人”。这就需要 `WHERE` 子句。

**语法格式：**

```sql
SELECT 列名列表 FROM 表名 WHERE 条件;
```

- `WHERE` 写在 `FROM` 后面。
- “条件”是一个**布尔表达式**，结果为“真（true）”的行才会被查出来。
- 可以把它理解成一个**筛子**：一行一行地过，条件成立的留下，不成立的丢掉。

### 4.1 比较运算符

`WHERE` 里最常用的就是比较运算符。

| 运算符      | 含义       | 示例                  |
|-------------|------------|-----------------------|
| `=`         | 等于       | `salary = 8000`       |
| `>`         | 大于       | `salary > 10000`      |
| `<`         | 小于       | `salary < 8000`       |
| `>=`        | 大于等于   | `salary >= 9500`      |
| `<=`        | 小于等于   | `salary <= 8000`      |
| `!=`        | 不等于     | `gender != '男'`      |
| `<>`        | 不等于     | `gender <> '男'`      |

> 💡 提示：`!=` 和 `<>` 完全等价，都表示“不等于”。`<>` 是标准 SQL 写法，`!=` 是大多数程序员更习惯的写法，两者随便用。
>
> ⚠️ 注意：判断“相等”用**一个等号 `=`**，不是编程语言里的 `==`！MySQL 里写 `==` 会报错。

**示例 1：查询工资大于 10000 的员工**

```sql
SELECT ename, salary FROM emp WHERE salary > 10000;
```

**执行结果：**

| ename | salary |
|-------|--------|
| 李四  | 12000  |
| 孙七  | 15000  |

**示例 2：查询所有女员工**

```sql
SELECT ename, gender, salary FROM emp WHERE gender = '女';
```

**执行结果：**

| ename | gender | salary |
|-------|--------|--------|
| 王五  | 女     | 9500   |
| 赵六  | 女     | 6000   |

> ⚠️ 注意：字符串值（如 `'女'`、`'男'`）必须用**单引号**括起来。数字（如 `10000`）不用加引号。

**示例 3：查询不是研发部（dept_id 不等于 1）的员工**

```sql
SELECT ename, dept_id FROM emp WHERE dept_id != 1;
-- 等价：WHERE dept_id <> 1;
```

**执行结果：**

| ename | dept_id |
|-------|---------|
| 王五  | 2       |
| 赵六  | 2       |
| 孙七  | 3       |

### 4.2 范围查询 BETWEEN ... AND ...

当条件是“在某个区间内”时，可以用 `BETWEEN ... AND ...`，比写两个 `>=` 和 `<=` 更简洁。

**语法格式：**

```sql
WHERE 列名 BETWEEN 下限 AND 上限;
```

- 含义：列值 **大于等于** 下限 **且 小于等于** 上限。
- **是闭区间（包含两端）**，等价于 `列名 >= 下限 AND 列名 <= 上限`。
- 下限必须写在前、上限写在后，写反了查不到数据。

**示例：查询工资在 8000 到 12000 之间（含 8000 和 12000）的员工**

```sql
SELECT ename, salary FROM emp WHERE salary BETWEEN 8000 AND 12000;
```

**执行结果（8000 和 12000 都被包含进来了）：**

| ename | salary |
|-------|--------|
| 张三  | 8000   |
| 李四  | 12000  |
| 王五  | 9500   |

它完全等价于：

```sql
SELECT ename, salary FROM emp WHERE salary >= 8000 AND salary <= 12000;
```

> 🕳️ 常见坑：`BETWEEN a AND b` 一定要 **a ≤ b**。如果写成 `BETWEEN 12000 AND 8000`（大的在前），结果会是**空集**，因为没有任何值能“同时大于等于 12000 又小于等于 8000”。

> 💡 提示：`BETWEEN ... AND ...` 也能用于**日期**。比如查 2020 年至 2021 年入职的：`WHERE join_date BETWEEN '2020-01-01' AND '2021-12-31'`。

### 4.3 集合查询 IN (...)

当条件是“等于这几个值中的任意一个”时，用 `IN`，比写一堆 `OR` 清爽得多。

**语法格式：**

```sql
WHERE 列名 IN (值1, 值2, 值3, ...);
```

- 含义：列值**等于括号里任意一个值**就算匹配。
- 等价于 `列名 = 值1 OR 列名 = 值2 OR ...`。

**示例：查询研发部（1）和财务部（3）的员工**

```sql
SELECT ename, dept_id FROM emp WHERE dept_id IN (1, 3);
```

**执行结果：**

| ename | dept_id |
|-------|---------|
| 张三  | 1       |
| 李四  | 1       |
| 孙七  | 3       |

它等价于（但啰嗦得多）：

```sql
SELECT ename, dept_id FROM emp WHERE dept_id = 1 OR dept_id = 3;
```

**取反：用 `NOT IN` 查“不在这些值里”的**

```sql
SELECT ename, dept_id FROM emp WHERE dept_id NOT IN (1, 3);
```

**执行结果（只剩 2 号部门的人）：**

| ename | dept_id |
|-------|---------|
| 王五  | 2       |
| 赵六  | 2       |

> 🕳️ 常见坑：`NOT IN` 遇到 `NULL` 会很诡异。如果 `IN` 的列表里包含 `NULL`（如 `NOT IN (1, NULL)`），整个条件可能永远返回“非真”，导致查不到任何数据。原因还是 `NULL` 的“未知”特性（见下一节）。所以**用 `NOT IN` 时务必保证列表里没有 NULL**。

### 4.4 NULL 的判断：IS NULL / IS NOT NULL

这是初学者**最容易踩的坑**之一，必须单独拎出来重点讲。

**核心规则：判断一个值是不是 `NULL`，必须用 `IS NULL` / `IS NOT NULL`，绝对不能用 `= NULL` 或 `!= NULL`。**

**为什么不能用 `= NULL`？**

因为 `NULL` 的语义是**“未知”**。在 SQL 的三值逻辑（true / false / **unknown**）里：

- `任何值 = NULL` 的结果不是 true 也不是 false，而是 **unknown（未知）**；
- `WHERE` 只保留结果为 **true** 的行，`unknown` 会被当作“不满足”丢弃。

用大白话讲：你问“某人的奖金等于‘未知’吗？”——这个问题本身就没法回答“是”或“否”，所以 MySQL 给你 `unknown`，于是一行都查不出来。

**反面示例（错误写法，查不到任何数据）：**

```sql
SELECT ename, bonus FROM emp WHERE bonus = NULL;
```

**执行结果：空集（Empty set），一行都没有！** 明明李四、赵六的 bonus 是 NULL，却查不出来。

**正确写法：用 `IS NULL`**

```sql
SELECT ename, bonus FROM emp WHERE bonus IS NULL;
```

**执行结果：**

| ename | bonus |
|-------|-------|
| 李四  | NULL  |
| 赵六  | NULL  |

**反向：查“有奖金”的（bonus 不为 NULL），用 `IS NOT NULL`**

```sql
SELECT ename, bonus FROM emp WHERE bonus IS NOT NULL;
```

**执行结果：**

| ename | bonus |
|-------|-------|
| 张三  | 1000  |
| 王五  | 2000  |
| 孙七  | 3000  |

> ⚠️ 注意（务必牢记）：
> - 判断为空：`IS NULL` ✅，`= NULL` ❌
> - 判断非空：`IS NOT NULL` ✅，`!= NULL` ❌、`<> NULL` ❌
>
> 这是 MySQL 笔试面试的高频考点，也是实际写 bug 的高发区。

### 4.5 逻辑运算符 AND / OR / NOT

当一个查询有**多个条件**时，用逻辑运算符把它们连起来。

| 逻辑运算符 | 等价符号 | 含义                                | 通俗理解         |
|------------|----------|-------------------------------------|------------------|
| `AND`      | `&&`     | 两个条件**都**成立才为真            | “并且 / 而且”    |
| `OR`       | `\|\|`   | 两个条件**有一个**成立就为真        | “或者”           |
| `NOT`      | `!`      | 对条件取反（真变假、假变真）        | “不 / 非”        |

> 💡 提示：`AND`/`OR`/`NOT` 是标准写法，可读性好，**强烈推荐用单词形式**。`&&`/`||`/`!` 是 MySQL 提供的等价符号，能用但不推荐（其它数据库未必支持，且容易和编程语言混淆）。

**示例 1：AND——查“男性”且“工资大于 10000”的员工（两个条件都要满足）**

```sql
SELECT ename, gender, salary FROM emp WHERE gender = '男' AND salary > 10000;
```

**执行结果（张三是男但工资 8000 不满足，被排除）：**

| ename | gender | salary |
|-------|--------|--------|
| 李四  | 男     | 12000  |
| 孙七  | 男     | 15000  |

**示例 2：OR——查“工资低于 7000”或“工资高于 14000”的员工（满足其一即可）**

```sql
SELECT ename, salary FROM emp WHERE salary < 7000 OR salary > 14000;
```

**执行结果：**

| ename | salary |
|-------|--------|
| 赵六  | 6000   |
| 孙七  | 15000  |

**示例 3：NOT——查“不是研发部”的员工**

```sql
SELECT ename, dept_id FROM emp WHERE NOT dept_id = 1;
-- 等价于 WHERE dept_id != 1;
```

**执行结果：**

| ename | dept_id |
|-------|---------|
| 王五  | 2       |
| 赵六  | 2       |
| 孙七  | 3       |

#### 优先级与括号

当 `AND` 和 `OR` 混用时，要特别小心**优先级**：**`NOT` > `AND` > `OR`**（NOT 最高，AND 次之，OR 最低）。也就是说，`AND` 会**先**于 `OR` 结合。

**易错示例：想查“研发部或市场部里，工资大于 9000 的人”**

如果不加括号这样写：

```sql
-- ❌ 含义被理解错了
SELECT ename, dept_id, salary FROM emp
WHERE dept_id = 1 OR dept_id = 2 AND salary > 9000;
```

由于 `AND` 优先级高，它实际被理解成：

```sql
WHERE dept_id = 1 OR (dept_id = 2 AND salary > 9000);
```

意思变成了“**所有研发部的人** 或者 **市场部里工资>9000的人**”，这不是我们要的。

**执行结果（张三 8000 也被查出来了，因为他属于研发部）：**

| ename | dept_id | salary |
|-------|---------|--------|
| 张三  | 1       | 8000   |
| 李四  | 1       | 12000  |
| 王五  | 2       | 9500   |

**正确写法：用括号 `()` 明确分组**

```sql
SELECT ename, dept_id, salary FROM emp
WHERE (dept_id = 1 OR dept_id = 2) AND salary > 9000;
```

**执行结果（张三 8000 被正确排除）：**

| ename | dept_id | salary |
|-------|---------|--------|
| 李四  | 1       | 12000  |
| 王五  | 2       | 9500   |

> 🕳️ 常见坑：**只要 `AND` 和 `OR` 同时出现，就老老实实加括号！** 不要去赌优先级，括号既能保证逻辑正确，又能让别人一眼看懂。这是写 SQL 的好习惯。

## 5. 模糊查询 LIKE

前面的比较都是“精确匹配”（等于、大于……）。但很多时候我们只记得“姓张的”“名字带五的”这种**部分信息**，这就需要**模糊查询 `LIKE`**，配合**通配符**使用。

**语法格式：**

```sql
WHERE 列名 LIKE '匹配模式';
```

两个通配符（核心中的核心）：

| 通配符 | 含义                       | 类比                       |
|--------|----------------------------|----------------------------|
| `%`    | 匹配**任意多个**字符（含0个）| 像“万能牌”，能顶任意长度   |
| `_`    | 匹配**任意单个**字符（恰好1个）| 像“占一个坑”，不多不少一个 |

> 💡 提示：记忆法——`%` 是“百分比”、量大，代表“任意多个”；`_` 是“一条下划线、一个空位”，代表“恰好一个”。

### 5.1 % 通配符：匹配任意多个字符

**示例 1：查询所有“张”姓员工（姓张，名字几个字都行）**

```sql
SELECT ename FROM emp WHERE ename LIKE '张%';
```

`'张%'` 表示“以‘张’开头，后面跟任意多个字符”。

**执行结果：**

| ename |
|-------|
| 张三  |

**示例 2：查询名字里**包含“五”**的员工（不管在开头、中间还是结尾）**

```sql
SELECT ename FROM emp WHERE ename LIKE '%五%';
```

`'%五%'` 表示“‘五’前面任意字符、后面也任意字符”，即只要名字里有“五”就匹配。

**执行结果：**

| ename |
|-------|
| 王五  |

**示例 3：查询名字以“三”**结尾**的员工**

```sql
SELECT ename FROM emp WHERE ename LIKE '%三';
```

**执行结果：**

| ename |
|-------|
| 张三  |

### 5.2 _ 通配符：匹配单个字符

**示例：查询名字是“两个字、且第二个字是‘四’”的员工**

```sql
SELECT ename FROM emp WHERE ename LIKE '_四';
```

`'_四'` 表示“**第一个位置是任意一个字符**，第二个位置必须是‘四’”——也就是名字**恰好两个字、以‘四’结尾**。

**执行结果：**

| ename |
|-------|
| 李四  |

> 🕳️ 常见坑：`_` 严格匹配**一个**字符，多一个少一个都不行。
> - `LIKE '_四'` 只能匹配“X四”这种两字名（如李四），匹配不到“张大四”这种三字名。
> - 如果想匹配“以四结尾、字数不限”，要用 `%`：`LIKE '%四'`。

### 5.3 % 和 _ 组合使用

两个通配符可以一起用，组合出更精确的模式。

**示例：查询“名字第一个字是‘王’，且总共两个字”的员工**

```sql
SELECT ename FROM emp WHERE ename LIKE '王_';
```

`'王_'` 表示“以‘王’开头，后面再跟**恰好一个**字符”。

**执行结果：**

| ename |
|-------|
| 王五  |

### 5.4 LIKE 的注意事项与坑

**坑 1：不带通配符的 LIKE，等同于精确匹配 `=`**

```sql
SELECT ename FROM emp WHERE ename LIKE '张三';
```

这没有用到 `%` 或 `_`，所以效果和 `WHERE ename = '张三'` 一模一样，只能查到完全等于“张三”的。

**坑 2：想查“真正的 % 或 _ 符号”怎么办？用转义**

如果某列的数据里**本身就含有 `%` 或 `_`** 这两个字符（比如折扣字段存了“50%”），直接 `LIKE '%50%%'` 会把 `%` 当通配符。这时要用 `ESCAPE` 指定转义符：

```sql
-- 查 discount 列中真正包含 "50%" 字符的行
SELECT * FROM some_table WHERE discount LIKE '%50\%%' ESCAPE '\';
```

`ESCAPE '\'` 告诉 MySQL：`\` 后面的那个 `%` 是普通字符，不是通配符。

> ⚠️ 注意：本套示例数据 `emp` 里没有这种带特殊符号的数据，此处只作了解，知道有这个机制即可。

**坑 3：以 % 开头的 LIKE 性能差**

`LIKE '%五%'` 和 `LIKE '%五'`（以 `%` 开头）会导致**索引失效**，MySQL 不得不逐行扫描整张表，数据量大时很慢。而 `LIKE '张%'`（`%` 在结尾）则能用上索引。所以：

- **能用 `'前缀%'` 就别用 `'%关键字%'`**；
- 真要做“包含”式全文检索，应考虑**全文索引（FULLTEXT）**或专门的搜索引擎（如 ElasticSearch），这是后话。

**坑 4：LIKE 大小写是否敏感？**

默认情况下，MySQL 的 `LIKE` 对**英文字母大小写不敏感**（取决于列的字符集排序规则 collation，如默认的 `utf8mb4_general_ci` 里 `ci` 就是 case-insensitive 不区分大小写）。例如 `LIKE 'a%'` 也能匹配到 `Apple`。如需严格区分大小写，可用 `LIKE BINARY 'a%'`。中文不涉及大小写问题，本章数据不受影响。

## 6. 综合演练

把本章学的串起来，看一个稍复杂的真实查询：

**需求：查询“研发部或市场部”里、“工资在 8000~13000 之间”、“有奖金”的男员工，显示姓名、工资、奖金，以及工资奖金合计（奖金为空算 0），并把合计列叫“月收入”。**

```sql
SELECT ename            AS 姓名,
       salary           AS 工资,
       bonus            AS 奖金,
       salary + IFNULL(bonus, 0) AS 月收入
FROM emp
WHERE dept_id IN (1, 2)            -- 研发部或市场部
  AND salary BETWEEN 8000 AND 13000 -- 工资区间（闭区间）
  AND bonus IS NOT NULL             -- 有奖金（非 NULL）
  AND gender = '男';                -- 男员工
```

逐个条件套到数据上分析：

- `dept_id IN (1,2)`：留下 张三、李四、王五、赵六；
- `salary BETWEEN 8000 AND 13000`：赵六 6000 出局 → 留下 张三(8000)、李四(12000)、王五(9500)；
- `bonus IS NOT NULL`：李四 bonus 为 NULL 出局 → 留下 张三、王五；
- `gender = '男'`：王五是女，出局 → 只剩 张三。

**执行结果：**

| 姓名 | 工资 | 奖金 | 月收入 |
|------|------|------|--------|
| 张三 | 8000 | 1000 | 9000   |

> 💡 提示：写复杂 `WHERE` 时，习惯把每个条件单独成行、`AND`/`OR` 放在行首对齐，可读性会好很多——就像上面那样。

## 7. 本章小结

- **DQL（SELECT）是最常用、最重要**的 SQL，只读不改，可放心练习。
- **完整骨架**：`SELECT ... FROM ... WHERE ... GROUP BY ... HAVING ... ORDER BY ... LIMIT`，书写顺序固定。本章深入了前三个：`SELECT`（查哪些列）、`FROM`（查哪张表）、`WHERE`（行过滤）。
- **基础查询**：
  - `SELECT *` 查全部列（**正式代码别用**，要显式写列名）；
  - 列别名用 `AS`（**可省略**），表别名同理（起了别名后原表名不能再用）。
- **去重 / 计算 / 空值**：
  - `DISTINCT` 去重，作用于**后面所有列的组合**；
  - `SELECT` 里可写算术表达式（如 `salary * 12`），记得**起别名**，且**不改原数据**；
  - `NULL` 表示“未知”，**任何值与 NULL 运算都得 NULL**，用 `IFNULL(列, 默认值)` 兜底。
- **条件查询 WHERE**：
  - 比较运算符 `> < >= <= =`，不等于用 `!=` 或 `<>`（相等用**一个** `=`）；
  - 区间用 `BETWEEN a AND b`（**闭区间**，a 必须 ≤ b）；
  - 集合用 `IN (...)` / `NOT IN (...)`；
  - **判空必须用 `IS NULL` / `IS NOT NULL`，绝不能用 `= NULL`**；
  - 多条件用 `AND` / `OR` / `NOT`，优先级 `NOT > AND > OR`，**混用时务必加括号**。
- **模糊查询 LIKE**：`%` 匹配任意多个字符，`_` 匹配单个字符；`'前缀%'` 能用索引，`'%关键字%'` 会全表扫描。

## 8. 常见面试 / 易错问答

**Q1：判断某列是否为空，能不能用 `WHERE 列 = NULL`？**
不能。`= NULL` 永远返回 `unknown`，查不到任何行。必须用 `IS NULL` / `IS NOT NULL`。根本原因是 `NULL` 表示“未知值”，参与比较运算结果是“未知”，而 `WHERE` 只保留结果为真的行。

**Q2：`!=` 和 `<>` 有区别吗？**
没有区别，都是“不等于”。`<>` 是标准 SQL 写法，`!=` 是程序员习惯写法，MySQL 两者都支持。

**Q3：`SELECT salary + bonus` 为什么有的行变成 NULL 了？怎么解决？**
因为该行 `bonus` 是 `NULL`，而任何数 `+ NULL = NULL`。解决：`salary + IFNULL(bonus, 0)`，把 NULL 当 0 处理。

**Q4：`BETWEEN 10 AND 20` 包含 10 和 20 吗？**
包含。它是**闭区间**，等价于 `>= 10 AND <= 20`。

**Q5：`DISTINCT a, b` 是只对 a 去重吗？**
不是。是对 `(a, b)` 这个**组合**整体去重，只有 a 和 b 都相同的行才会被合并。

**Q6：`WHERE dept_id = 1 OR dept_id = 2 AND salary > 9000` 的结果对吗？**
逻辑容易错。因为 `AND` 优先级高于 `OR`，它等于 `dept_id = 1 OR (dept_id = 2 AND salary > 9000)`。若本意是“(1部门 或 2部门) 且 工资>9000”，必须加括号：`(dept_id = 1 OR dept_id = 2) AND salary > 9000`。

**Q7：`LIKE '_四'` 和 `LIKE '%四'` 有什么区别？**
`_四` 匹配“恰好两个字、以四结尾”（如李四），`_` 严格代表一个字符；`%四` 匹配“以四结尾、长度不限”（如李四、张大四都行），`%` 代表任意多个字符。

**Q8：为什么尽量别用 `LIKE '%xxx%'`？**
以 `%` 开头会导致索引失效、全表扫描，大数据量下性能很差。能用 `'前缀%'` 就用前缀匹配；需要“包含”式搜索时考虑全文索引或专业搜索引擎。

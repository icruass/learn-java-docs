# SQL 概述、通用语法与四大分类（DDL / DML / DQL / DCL）

> **本章导读**
>
> 前两章我们安装好了 MySQL、连上了数据库、也认识了“数据库 / 表 / 行 / 列”这些基本概念。但到目前为止，我们还只是“看客”——真正要让数据库干活，靠的是一门语言：**SQL**。
>
> 这一章就是整套教程的“语言总纲”。它本身**几乎不教你具体怎么建表、怎么查数据**（那是后面几章的事），而是先回答三个根本问题：
>
> 1. **SQL 是什么？** 它为什么能成为操作几乎所有关系型数据库的“普通话”？
> 2. **怎么写 SQL 才算合法？** 大小写、分号、空格、注释……这些“书写规矩”不弄清楚，后面写任何语句都会踩坑。
> 3. **SQL 一共能干哪些事？** 这就是著名的“四大分类”：DDL、DML、DQL、DCL。把它们分清楚，你脑子里就有了一张地图——后面每学一条语句，你都知道它属于哪一类、操作的是“结构”还是“数据”还是“权限”。
>
> 可以把本章理解为：**先发一张地图和一本“交通规则手册”，再上路开车。** 地图就是四大分类，交通规则就是通用语法。读完本章，你不一定会写多复杂的 SQL，但你一定能看懂任何一条 SQL“是哪一类、大概在干嘛、写得合不合规矩”。
>
> 本章会用到的示例库就是全套教程统一的 `db_learn`（部门表 `dept`、员工表 `emp`），后续章节会反复出现，请务必先建好它们（建表语句在最后“附录”中给出，本章正文只做演示引用）。

## 一、SQL 是什么

### 1.1 一句话定义

**SQL = Structured Query Language = 结构化查询语言。**

它是一门**专门用来操作关系型数据库**的语言。注意三个关键词：

- **Structured（结构化）**：关系型数据库里的数据是“一行一行、一列一列”规整存放的（像 Excel 表格），这种规整结构就叫“结构化数据”。SQL 就是为操作这种结构化数据而生的。
- **Query（查询）**：虽然名字里只有“查询”，但其实 SQL 远不止查询，它还能建库建表、增删改数据、管理权限。名字是历史遗留，别被它骗了。
- **Language（语言）**：它是一门“语言”，有自己的语法规则，你“说”给数据库听，数据库就照做。

> 💡 **提示：怎么读？**
> SQL 有两种常见读法，都对：一种逐字母读 “S-Q-L”；另一种读作 “sequel”（/ˈsiːkwəl/，谐音“C口”）。面试和日常交流随便用哪种都行。

### 1.2 一个最直观的类比

把数据库想象成一个**巨大的、带管理员的仓库**：

| 现实世界 | 数据库世界 |
| --- | --- |
| 仓库 | 数据库服务器（MySQL） |
| 仓库里的一个个房间 | 一个个数据库（database / schema） |
| 房间里的货架 | 表（table） |
| 货架上的一格 | 一行记录（row） |
| 你写给管理员的“取货 / 上货 / 改标签”的纸条 | 一条 SQL 语句 |

你不能自己跑进仓库乱翻，你只能**写纸条（SQL）交给管理员（数据库引擎）**，管理员看懂后帮你执行，再把结果递给你。SQL 就是你和数据库之间唯一的“沟通语言”。

### 1.3 为什么是“统一语言”——一次学会，到处能用

这是 SQL 最大的价值所在。市面上的关系型数据库很多：

- **MySQL**（本教程主角，开源、互联网公司最常用）
- **Oracle**（商业、银行 / 电信等大型企业常用）
- **SQL Server**（微软出品）
- **PostgreSQL**（开源、功能强大）
- **SQLite**（轻量级，手机 App、小工具常用）
- ……

它们底层实现千差万别，但**对外说的“话”大体是同一种——SQL**。

这意味着：你今天在 MySQL 上学会的 `SELECT * FROM emp;`，拿到 Oracle、SQL Server 上，**绝大多数情况一字不改也能跑**。这就是“统一语言”的威力——**学一次，走天下**。

> 💡 **提示：SQL 之于数据库，就像“普通话”之于全国各地。**
> 各地有方言（后面会讲），但大家都会说普通话，所以来自天南海北的人能用普通话顺畅交流。SQL 就是关系型数据库界的普通话。

## 二、SQL 标准与方言（为什么不同数据库语法有细微差异）

### 2.1 有“国标”：SQL 是有标准的

SQL 不是某家公司私有的，它是有**国际标准**的。标准由 ISO（国际标准化组织）和 ANSI（美国国家标准学会）共同维护，历史上有 SQL-92、SQL:1999、SQL:2003、SQL:2008…… 等多个版本（你不需要背这些年份，知道“有国标”即可）。

正因为有这套标准，各家数据库才能在“主干语法”上保持一致——这就是 1.3 节里“一次学会到处能用”的根本保证。

### 2.2 有“方言”：各厂商的私有扩展

但是，标准往往“管得不够细”或者“更新慢”，而每家数据库厂商为了让自己的产品更好用、更有竞争力，都会在标准之外**加一些自己独有的功能或写法**。这些“标准之外的私有部分”，就叫做**方言（Dialect）**。

于是就出现了这样的局面：

- **主干（标准 SQL）**：人人都遵守，通用。
- **枝叶（方言）**：各家不同，换数据库就可能要改。

> 💡 **类比：** 普通话是标准，但北京人说“倍儿好”、东北人说“贼好”、四川人说“巴适”。这些方言词外地人不一定懂，换个地方就得换说法。SQL 方言也是同理。

### 2.3 几个能让你“立刻有感觉”的方言例子

光说概念太空，看几个真实差异（**不用记，感受“原来真的会不一样”就够了**）：

| 需求 | MySQL 写法 | Oracle 写法 | SQL Server 写法 |
| --- | --- | --- | --- |
| 查询前 3 行 | `... LIMIT 3` | `... WHERE ROWNUM <= 3` | `SELECT TOP 3 ...` |
| 自动增长主键 | `AUTO_INCREMENT` | 序列 `SEQUENCE` + 触发器 | `IDENTITY(1,1)` |
| 字符串拼接 | `CONCAT('a','b')` | `'a' \|\| 'b'` | `'a' + 'b'` |
| 取系统当前时间 | `NOW()` | `SYSDATE` | `GETDATE()` |

例如同样是“查工资最高的前 3 名员工”：

```sql
-- MySQL 方言：用 LIMIT
SELECT ename, salary
FROM emp
ORDER BY salary DESC
LIMIT 3;
```

```sql
-- Oracle 方言（同样需求，写法完全不同）：用 ROWNUM
SELECT ename, salary
FROM (SELECT ename, salary FROM emp ORDER BY salary DESC)
WHERE ROWNUM <= 3;
```

**执行结果（在 db_learn 上跑 MySQL 那条）：**

| ename | salary |
| --- | --- |
| 孙七 | 15000 |
| 李四 | 12000 |
| 王五 | 9500 |

> ⚠️ **注意：本教程默认讲的是 MySQL 方言。**
> 后面我们会用到 `LIMIT`、`AUTO_INCREMENT`、`# 注释`、反引号 `` `字段名` `` 等，这些有的是标准、有的是 MySQL 方言。我会在涉及方言时尽量提醒你“这是 MySQL 特有的”，这样你将来换数据库时心里有数。
>
> 🕳️ **常见坑：把 MySQL 写法当成“SQL 通用写法”。**
> 很多初学者在 MySQL 上写惯了 `LIMIT`，到了 Oracle 上一跑就报错，还一脸懵：“我语法没错啊？”——其实是用了方言。记住：**通用的才能到处跑，方言只在自家好使。**

## 三、SQL 通用语法规则（书写规矩）

这一节是“交通规则”。规则不多，但每一条都会贯穿你今后写的每一句 SQL。

### 3.1 可以单行写，也可以多行写

一条 SQL 语句，写成一行可以，写成多行也完全没问题。**数据库不看你换不换行，它看的是“分号在哪结束”。**

```sql
-- 写成一行：完全合法
SELECT id, ename, salary FROM emp WHERE dept_id = 1;
```

```sql
-- 写成多行：同样合法，而且更清晰（推荐复杂语句这样写）
SELECT id,
       ename,
       salary
FROM   emp
WHERE  dept_id = 1;
```

上面两条语句**含义完全相同，执行结果也完全相同**：

| id | ename | salary |
| --- | --- | --- |
| 1 | 张三 | 8000 |
| 2 | 李四 | 12000 |

> 💡 **提示：复杂语句强烈建议多行 + 缩进。**
> 简单的 `SELECT * FROM emp;` 一行就够；但当语句长到带 `WHERE`、`GROUP BY`、`ORDER BY`、多表连接时，把每个子句单独成行、对齐缩进，可读性会天差地别。代码是给人看的，机器不挑，但你的同事（和半年后的你自己）会感谢你。

### 3.2 每条语句以分号 `;` 结束

分号 `;` 是一条 SQL 语句的**“句号”**，告诉数据库：“这句话说完了，可以执行了。”

```sql
SELECT * FROM dept;     -- 第一条语句，到分号结束
SELECT * FROM emp;      -- 第二条语句，又到分号结束
```

为什么需要分号？因为你可能一次性发多条语句过去，数据库得知道**“从哪到哪算一句”**。分号就是那个分界点。

> 🕳️ **常见坑：忘写分号导致“假死”。**
> 在命令行客户端（如 `mysql>` 提示符下）敲了语句却忘了分号回车，你会看到提示符变成 `->`，光标停在那里好像卡住了——其实它在等你把这句“说完”（等分号）。这时只要补一个 `;` 再回车即可：
>
> ```bash
> mysql> SELECT * FROM emp
>     -> ;          # 这里补上分号，语句才真正执行
> ```
>
> 💡 **提示：**在图形客户端（Navicat、DataGrip、DBeaver、SQLyog 等）里，如果一次只执行一条语句，**省略结尾分号通常也能跑**；但只要你一次执行多条，分号就是**强制**的。所以养成“每句都加分号”的好习惯，绝不吃亏。

### 3.3 不区分大小写，但关键字建议大写

**SQL 关键字不区分大小写。** 下面这三条语句，对 MySQL 来说**完全一样**：

```sql
SELECT * FROM emp;
select * from emp;
SeLeCt * FrOm EmP;
```

它们的执行结果一模一样（返回 emp 表全部 5 行）。

那为什么大家约定俗成**关键字用大写**？

- **一眼区分“命令”和“数据”**：`SELECT`、`FROM`、`WHERE` 大写，表名 `emp`、字段名 `salary` 小写，读起来层次分明。
- **行业惯例 / 显专业**：几乎所有教材、文档、规范都这么写。

```sql
-- 推荐风格：关键字大写，库名/表名/字段名小写
SELECT ename, salary
FROM   emp
WHERE  salary > 9000;
```

> ⚠️ **注意：一个重要的“例外”——数据 / 标识符不一定不区分大小写！**
>
> “不区分大小写”指的是**关键字**（`SELECT`、`FROM` 这些）。但有两个东西要小心：
>
> 1. **存进去的数据值**：默认排序规则下，`WHERE ename = '张三'` 与值是否大小写敏感，取决于该列的**字符集与排序规则（collation）**。英文字母比如 `'abc'` 和 `'ABC'`，在 MySQL 默认的 `utf8mb4_general_ci`（ci = case insensitive，不区分大小写）下会被当作相等；如果用了 `_cs`（case sensitive）或 `_bin` 排序规则，则区分大小写。
> 2. **库名 / 表名**：在 **Linux** 服务器上，MySQL 的库名、表名**默认区分大小写**（因为底层是文件，Linux 文件名区分大小写）；在 **Windows** 上默认不区分。所以 `SELECT * FROM Emp;` 在 Windows 能跑，搬到 Linux 可能就报“表不存在”。
>
> 🕳️ **常见坑：** 本地 Windows 开发一切正常，部署到 Linux 服务器后报 `Table 'db_learn.Emp' doesn't exist`——十有八九就是表名大小写问题。**统一规范：库名、表名、字段名一律小写**，可一劳永逸避开此坑。

### 3.4 空格与缩进：随便加，但要加得“有意义”

SQL 中，标识符 / 关键字之间的**空格、换行、Tab 缩进，数据库一律当作“一个分隔符”处理**，多加几个、少加几个（只要至少有一个）都不影响结果。

```sql
-- 下面两条等价（一个空格 vs 一堆空格 + 换行）
SELECT ename FROM emp;

SELECT      ename
FROM        emp;
```

所以空格的作用**不是给机器看的，是给人看的**——用它来对齐、缩进、分组，让语句更易读。

> ⚠️ **注意：有些地方“不能乱加空格”。**
> - 函数名和它的括号之间，标准上可以有空格（如 `COUNT (*)`），但**强烈建议不要加**，写成 `COUNT(*)`，更清晰也更兼容。
> - 字符串**引号内部**的空格是**数据本身**，加了就改变了数据！`'张三'` 和 `'张三 '`（后面多个空格）是**两个不同的值**。
>
> ```sql
> -- 这两个查的不是一回事！
> SELECT * FROM emp WHERE ename = '张三';     -- 能查到张三
> SELECT * FROM emp WHERE ename = '张三 ';    -- 末尾多空格，查不到
> ```

## 四、SQL 的三种注释

注释 = 写给**人**看的说明文字，**数据库执行时会直接跳过**。注释能帮你解释“这段 SQL 在干嘛”，也能在调试时临时“屏蔽”掉某些语句。

MySQL 支持**三种**注释写法：

### 4.1 单行注释：`-- 注释内容`（标准 SQL，⚠️ 横杠后必须有空格）

```sql
-- 这是一行注释，查询所有员工
SELECT * FROM emp;

SELECT ename, salary FROM emp;  -- 注释也可以写在语句末尾
```

> 🕳️ **超高频坑：`--` 后面必须跟一个空格（或控制字符），否则不被当作注释！**
>
> 这是标准 SQL 的硬性规定。看对比：
>
> ```sql
> -- 正确：横杠、横杠、空格、再写文字
> SELECT 1;
>
> --错误：横杠后紧贴文字，MySQL 不认为这是注释，会报语法错误
> SELECT 1;
> ```
>
> 第二种写法里 `--错误...` 会被解析成语句的一部分，导致 `You have an error in your SQL syntax`。**口诀：`--` 之后，先敲个空格。**

### 4.2 单行注释：`# 注释内容`（⚠️ MySQL 特有的方言）

```sql
# 这也是单行注释（MySQL 特有），# 后面有没有空格都行
SELECT * FROM dept;

SELECT * FROM dept;  # 也能跟在语句后面
```

> ⚠️ **注意：`#` 注释是 MySQL 方言，不是标准 SQL！**
> 它写起来方便（不强制要空格），但**只有 MySQL 认**。如果你的 SQL 将来可能要在 Oracle、SQL Server 等其他数据库上运行，**请改用标准的 `--` 注释**，以保证可移植性。

### 4.3 多行注释：`/* 注释内容 */`（标准 SQL）

用于注释**一大段**内容，可以跨多行。以 `/*` 开始，以 `*/` 结束。

```sql
/*
  这是一段多行注释：
  下面这条 SQL 用于查询研发部（dept_id = 1）的所有员工，
  作者：张三   日期：2026-06-07
*/
SELECT * FROM emp WHERE dept_id = 1;

SELECT id, /* 也可以写在语句中间，把某一小段挡住 */ ename FROM emp;
```

> 💡 **提示：用多行注释“临时禁用”一段 SQL。**
> 调试时想暂时不执行某几行，又不想删掉，用 `/* ... */` 把它们框起来即可，省得来回删了又写。
>
> 🕳️ **常见坑：多行注释不能嵌套。**
> ```sql
> /* 外层 /* 内层 */ 还想继续注释 */   -- ❌ 出错！
> ```
> MySQL 遇到**第一个** `*/` 就认为注释结束了，后面的 `还想继续注释 */` 会被当成 SQL 代码而报错。**别在多行注释里再套多行注释。**

### 4.4 三种注释速查表

| 写法 | 类型 | 是否标准 SQL | 关键注意点 |
| --- | --- | --- | --- |
| `-- 内容` | 单行 | ✅ 标准 | `--` 后**必须有空格** |
| `# 内容` | 单行 | ❌ MySQL 方言 | 换库可能不识别，慎用 |
| `/* 内容 */` | 多行 | ✅ 标准 | **不能嵌套** |

## 五、SQL 四大分类（核心地图）

终于到了本章最重要的部分。SQL 语句虽然五花八门，但按**“它到底在操作什么”**，可以干净利落地分成**四大类**。把这四类搞清楚，你今后看任何一条 SQL，都能瞬间定位它“属于哪一类、在动哪样东西”。

先记住这张“四口诀”——按**操作对象**区分：

- **DDL** 管 **“结构”**（库长什么样、表有哪些列）
- **DML** 管 **“数据”** 的增删改（往表里塞数据、改数据、删数据）
- **DQL** 管 **“数据”** 的查询（把数据读出来看）
- **DCL** 管 **“权限 / 用户”**（谁能干什么）

下面逐一展开。

### 5.1 DDL —— 数据定义语言（Data Definition Language）

**作用：定义和管理“数据库对象的结构”**，比如创建/修改/删除 数据库、表、视图、索引等。**它动的是“骨架”，不是“数据”。**

**核心关键字：**

| 关键字 | 含义 | 通俗理解 |
| --- | --- | --- |
| `CREATE` | 创建 | 盖房子 / 打货架 |
| `ALTER` | 修改 | 给房子加扇窗、给货架加一格 |
| `DROP` | 删除 | 把整栋房子 / 整个货架拆掉 |
| `TRUNCATE` | 清空（表） | 货架还在，但货全倒空、编号归零 |

**示例：**

```sql
-- CREATE：创建数据库
CREATE DATABASE db_learn;

-- CREATE：创建一张部门表（定义结构：有哪些列、什么类型）
CREATE TABLE dept (
    id        INT PRIMARY KEY AUTO_INCREMENT,  -- 部门编号
    dept_name VARCHAR(20),                      -- 部门名称
    loc       VARCHAR(20)                       -- 所在城市
);

-- ALTER：给 dept 表新增一列“电话”
ALTER TABLE dept ADD phone VARCHAR(20);

-- DROP：删除整张表（连结构带数据一起没了）
DROP TABLE dept;

-- TRUNCATE：清空 emp 表的所有数据，但保留表结构（且自增主键归零）
TRUNCATE TABLE emp;
```

> ⚠️ **注意：DDL 操作通常“不可回滚”，下手要谨慎！**
> 在 MySQL 中，`CREATE`、`ALTER`、`DROP`、`TRUNCATE` 等 DDL 会**隐式提交（implicit commit）**，意味着你**没法靠 `ROLLBACK` 撤销**。`DROP TABLE emp;` 一旦执行，表就真没了。生产环境执行 DDL 前，请务必三思、最好先备份。
>
> 🕳️ **常见坑：`TRUNCATE` vs `DELETE` 傻傻分不清。**
> 两者都能“清空表里的数据”，但本质不同：
> | 对比项 | `TRUNCATE TABLE emp;` | `DELETE FROM emp;` |
> | --- | --- | --- |
> | 所属分类 | **DDL** | **DML** |
> | 原理 | 直接“摧毁重建”表，几乎不逐行处理 | 一行一行地删 |
> | 速度 | 快（数据多时明显快） | 慢 |
> | 能否带 `WHERE` 条件删部分 | ❌ 不能，只能全清 | ✅ 能，可只删符合条件的行 |
> | 自增主键 | **重置归零**（下次从 1 开始） | **不重置**（接着上次的值继续） |
> | 能否 `ROLLBACK` 撤销 | ❌ 不能 | ✅ 能（在事务中） |
>
> 一句话：要**带条件删、想能反悔**用 `DELETE`；要**整表秒清、不在乎主键归零**用 `TRUNCATE`。

### 5.2 DML —— 数据操作语言（Data Manipulation Language）

**作用：对表里的“数据”进行增、改、删。** 注意——它操作的是**表中的记录（行）**，**不动表结构**。

**核心关键字：**

| 关键字 | 含义 | 通俗理解 |
| --- | --- | --- |
| `INSERT` | 插入（增） | 往货架上放一件新货 |
| `UPDATE` | 更新（改） | 把货架上某件货的标签改一下 |
| `DELETE` | 删除（删） | 把货架上某件货拿走 |

**示例（在 emp 表上操作）：**

```sql
-- INSERT：新增一名员工
INSERT INTO emp (ename, gender, salary, join_date, dept_id, bonus)
VALUES ('周八', '男', 7000, '2023-03-01', 1, 500);

-- UPDATE：给“张三”涨工资到 8500（⚠️ 一定要带 WHERE！）
UPDATE emp SET salary = 8500 WHERE ename = '张三';

-- DELETE：删除“周八”这条记录（⚠️ 一定要带 WHERE！）
DELETE FROM emp WHERE ename = '周八';
```

**执行后 emp 表状态变化示意（以 INSERT 后为例）：**

| id | ename | gender | salary | join_date | dept_id | bonus |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 张三 | 男 | 8000 | 2020-01-10 | 1 | 1000 |
| 2 | 李四 | 男 | 12000 | 2019-03-15 | 1 | NULL |
| 3 | 王五 | 女 | 9500 | 2021-06-01 | 2 | 2000 |
| 4 | 赵六 | 女 | 6000 | 2022-09-20 | 2 | NULL |
| 5 | 孙七 | 男 | 15000 | 2018-11-05 | 3 | 3000 |
| **6** | **周八** | **男** | **7000** | **2023-03-01** | **1** | **500** |

> 🕳️ **致命坑：`UPDATE` / `DELETE` 不写 `WHERE`，全表遭殃！**
>
> ```sql
> UPDATE emp SET salary = 0;   -- 💀 没有 WHERE：所有人的工资全被改成 0！
> DELETE FROM emp;             -- 💀 没有 WHERE：整张表的数据全没了！
> ```
>
> 这是新手（乃至老手）最容易酿成事故的地方。养成习惯：**写 `UPDATE` / `DELETE` 时，先把 `WHERE` 条件写好，再回头补前面的部分。**
>
> 💡 **提示：开启“安全模式”防误删。** MySQL 客户端可设置 `SET SQL_SAFE_UPDATES = 1;`，开启后，没带 `WHERE`（或 `WHERE` 未用到键/索引）的 `UPDATE`/`DELETE` 会被直接拒绝执行，给你一道“保险”。

### 5.3 DQL —— 数据查询语言（Data Query Language）

**作用：查询数据，把表里的数据“读”出来给你看。** 关键字只有一个核心：**`SELECT`**（常配 `FROM`、`WHERE`、`GROUP BY`、`ORDER BY`、`LIMIT` 等）。

```sql
-- 最简单的查询：查 emp 表所有行所有列
SELECT * FROM emp;

-- 带条件查询：工资大于 9000 的员工的姓名和工资
SELECT ename, salary
FROM   emp
WHERE  salary > 9000
ORDER BY salary DESC;
```

**第二条的执行结果：**

| ename | salary |
| --- | --- |
| 孙七 | 15000 |
| 李四 | 12000 |
| 王五 | 9500 |

> 💡 **提示：为什么把 DQL 单独拎出来讲？**
> 严格按学术 / 标准定义，**`SELECT` 其实属于 DML 的一部分**（它也是“操作数据”嘛）。但在实际工作中，**“查询”是使用频率最高、内容最丰富、最值得花时间钻研的一块**——`WHERE` 条件、聚合函数、分组、排序、多表连接、子查询…… 复杂度远超增删改。
>
> 所以教学和工程实践中，习惯把 `SELECT` 从 DML 里**单独剥离**出来，称为 **DQL**，给予“一等公民”待遇。本套教程后续会用**整整好几章**专门讲 DQL，足见其重要性。你只要记住：**“DQL 本是 DML 的一部分，因太重要而被单列。”**
>
> ⚠️ **注意：`SELECT` 不修改任何数据。** 它是“只读”的，无论你查多少次，表里的数据都纹丝不动。这点和 DML 的增删改有本质区别。

### 5.4 DCL —— 数据控制语言（Data Control Language）

**作用：管理“数据库用户”和“权限”——即控制“谁，能对什么，做什么”。** 这是 DBA（数据库管理员）的活儿，普通开发用得少，但必须知道它的存在。

**核心关键字：**

| 关键字 | 含义 | 通俗理解 |
| --- | --- | --- |
| `GRANT` | 授予权限 | 发一张“门禁卡 / 通行证” |
| `REVOKE` | 收回权限 | 把通行证收回作废 |

（广义上 `CREATE USER`、`DROP USER`、`ALTER USER`、设置密码等用户管理操作也常被归入 DCL 范畴。）

**示例：**

```sql
-- 创建一个新用户 zhangsan，密码 123456，仅允许从本机登录
CREATE USER 'zhangsan'@'localhost' IDENTIFIED BY '123456';

-- GRANT：授予 zhangsan 对 db_learn 库的“查询 + 插入”权限
GRANT SELECT, INSERT ON db_learn.* TO 'zhangsan'@'localhost';

-- REVOKE：收回 zhangsan 的“插入”权限（以后他只能查，不能插了）
REVOKE INSERT ON db_learn.* FROM 'zhangsan'@'localhost';

-- 刷新权限，使更改立即生效（MySQL 习惯做法）
FLUSH PRIVILEGES;
```

**效果示意：**

| 操作前 | 操作后（REVOKE INSERT 之后） |
| --- | --- |
| zhangsan 能 `SELECT`、能 `INSERT` | zhangsan 只能 `SELECT`，`INSERT` 会被拒绝 |

> 💡 **提示：DCL 是“安全 / 运维”范畴。**
> 作为应用开发者，你日常更多是用一个**已经配好权限的账号**去连数据库写业务 SQL，很少亲手敲 `GRANT`/`REVOKE`。但理解它能帮你看懂“为什么我这个账号执行某条 SQL 会报 `command denied`（权限不足）”——多半就是 DCL 层面没给你对应权限。
>
> ⚠️ **注意：** 不同 MySQL 版本里，创建用户和授权的语法细节略有差异（比如旧版可在 `GRANT` 时顺带创建用户，新版推荐先 `CREATE USER` 再 `GRANT`）。这部分会在后续“用户与权限管理”章节细讲，此处只需建立“DCL 管权限”的概念。

## 六、四大分类总结对照表（务必记牢）

这张表是本章的“浓缩精华”，建议截图 / 抄下来反复看：

| 分类 | 英文全称 | 中文 | 核心关键字 | 操作对象 | 一句话用途 |
| --- | --- | --- | --- | --- | --- |
| **DDL** | Data **Definition** Language | 数据**定义**语言 | `CREATE`、`ALTER`、`DROP`、`TRUNCATE` | 数据库、表等**结构（骨架）** | 定义和管理库 / 表的结构 |
| **DML** | Data **Manipulation** Language | 数据**操作**语言 | `INSERT`、`UPDATE`、`DELETE` | 表中的**数据（记录 / 行）** | 对表里的数据进行增、改、删 |
| **DQL** | Data **Query** Language | 数据**查询**语言 | `SELECT` | 表中的**数据（只读）** | 把数据查询出来（本属 DML，因重要而单列） |
| **DCL** | Data **Control** Language | 数据**控制**语言 | `GRANT`、`REVOKE` | **用户与权限** | 控制谁能对数据库做什么 |

**记忆口诀：**

> **D-定义改结构（DDL），M-改数据增删改（DML），Q-查询只读取（DQL），C-控制管权限（DCL）。**

或者更短：**“DDL 改骨架，DML 改血肉，DQL 只看不动，DCL 管门禁。”**

> 💡 **提示：怎么快速判断一条 SQL 属于哪一类？**
> 看它的**第一个关键字（动词）**就够了：
> - `CREATE` / `ALTER` / `DROP` / `TRUNCATE` → **DDL**
> - `INSERT` / `UPDATE` / `DELETE` → **DML**
> - `SELECT` → **DQL**
> - `GRANT` / `REVOKE` → **DCL**

## 七、本章小结

- **SQL（Structured Query Language，结构化查询语言）** 是操作关系型数据库的**统一语言**，“学一次，走天下”。
- SQL 有**国际标准**，保证主干语法通用；各厂商又有**方言（私有扩展）**，导致不同数据库细节有差异（如 MySQL 用 `LIMIT`、Oracle 用 `ROWNUM`）。本教程默认讲 **MySQL 方言**。
- **通用语法规则**：
  - 可单行可多行；**以分号 `;` 结束**一条语句。
  - **关键字不区分大小写**，但**建议大写**；注意库名 / 表名在 Linux 上区分大小写——**统一小写**最稳妥。
  - 空格 / 缩进随便加（至少一个分隔），主要用于**提升可读性**；但**引号内的空格属于数据本身**。
- **三种注释**：
  - `-- 内容`（标准，**`--` 后必须有空格**）
  - `# 内容`（**MySQL 方言**，慎用）
  - `/* 内容 */`（标准多行，**不能嵌套**）
- **四大分类（核心）**：
  - **DDL**（`CREATE`/`ALTER`/`DROP`/`TRUNCATE`）—— 操作**结构**，通常不可回滚。
  - **DML**（`INSERT`/`UPDATE`/`DELETE`）—— 操作**数据**，改增删时**务必带 `WHERE`**。
  - **DQL**（`SELECT`）—— **查询**数据，只读；本属 DML，因重要而单列，是后续学习重点。
  - **DCL**（`GRANT`/`REVOKE`）—— 管理**用户与权限**。
- 判断归类：**看第一个关键字（动词）即可。**

## 八、常见面试 / 易错问答

**Q1：SQL 的四大分类是什么？各自的关键字？**
A：DDL（`CREATE`/`ALTER`/`DROP`/`TRUNCATE`，操作结构）、DML（`INSERT`/`UPDATE`/`DELETE`，操作数据）、DQL（`SELECT`，查询数据）、DCL（`GRANT`/`REVOKE`，控制权限）。

**Q2：`DELETE`、`TRUNCATE`、`DROP` 三者的区别？**
A：
- `DELETE`（DML）：逐行删数据，可带 `WHERE` 删部分，可回滚，自增不重置，表结构还在。
- `TRUNCATE`（DDL）：整表清空，不能带 `WHERE`，不可回滚，自增重置归零，表结构还在，速度快。
- `DROP`（DDL）：**连表结构带数据一起删掉**，表彻底消失，不可回滚。
- 一句话：删数据用 `DELETE`/`TRUNCATE`，删表用 `DROP`。

**Q3：为什么 `SELECT` 被单独叫做 DQL，而不是归到 DML？**
A：从标准定义看 `SELECT` 本属 DML（数据操作的一种）；但因查询是实际使用中最频繁、最复杂、最核心的部分，工程和教学习惯把它单独剥离出来称为 DQL，以示重视。

**Q4：`--` 注释为什么有时不生效 / 报错？**
A：标准 SQL 要求 `--` 后必须紧跟一个**空格**（或控制字符）才被识别为注释。写成 `--内容`（紧贴）不会被当作注释，从而引发语法错误。

**Q5：同一条 SQL 在 MySQL 能跑，换到 Oracle 报错，为什么？**
A：很可能用到了 **MySQL 方言**（如 `LIMIT`、`#` 注释、`AUTO_INCREMENT`、反引号等），这些不属于通用标准，其他数据库不识别。需改用目标数据库对应的语法。

**Q6：SQL 关键字到底区分不区分大小写？**
A：**关键字不区分**（`SELECT` 与 `select` 等价）。但要注意：①列值是否区分大小写取决于排序规则（collation）；②库名 / 表名在 Linux 下默认区分大小写、Windows 下不区分。规范建议：关键字大写，库 / 表 / 字段名一律小写。

# 多表查询：笛卡尔积、内连接、外连接与子查询

> **本章导读**
>
> 第 10 章我们把数据合理地拆成了多张表（部门一张、员工一张），第 11 章用范式论证了这样拆的道理。但拆开之后，查询时往往需要**把它们再拼回去**：「查每个员工**和他所在部门的名字**」——员工姓名在 `emp` 表，部门名字在 `dept` 表，一张表查不全。这就需要**多表查询（连接查询）**。
>
> 本章是 DQL 的高级篇，也是 SQL 的**重中之重、面试必考**，要讲透：
> 1. 直接把两张表放一起会产生**笛卡尔积**这个「错误起点」，理解它才能理解连接；
> 2. **内连接**（隐式 / 显式）——查两表都匹配上的数据；
> 3. **外连接**（左 / 右）——以某一张表为主，把没匹配上的也查出来；
> 4. **子查询**——把一个查询的结果当作另一个查询的条件或数据源，按结果形态分三种用法；
> 5. 三道综合练习把上面全部串起来。
>
> 本章沿用统一示例库 `db_learn` 的 `dept`（部门）和 `emp`（员工）两表，先把它们建好。

## 〇、准备示例数据

```sql
-- 部门表
CREATE TABLE dept (
  id        INT PRIMARY KEY AUTO_INCREMENT,
  dept_name VARCHAR(20),
  loc       VARCHAR(20)
);
INSERT INTO dept (dept_name, loc) VALUES
  ('研发部','北京'), ('市场部','上海'), ('财务部','广州'),
  ('行政部','深圳');   -- 行政部故意先不放人，用于演示外连接

-- 员工表
CREATE TABLE emp (
  id        INT PRIMARY KEY AUTO_INCREMENT,
  ename     VARCHAR(20),
  gender    CHAR(1),
  salary    DOUBLE,
  join_date DATE,
  dept_id   INT,
  bonus     DOUBLE,
  CONSTRAINT fk_emp_dept FOREIGN KEY (dept_id) REFERENCES dept(id)
);
INSERT INTO emp (ename, gender, salary, join_date, dept_id, bonus) VALUES
  ('张三','男',8000, '2020-01-10', 1, 1000),
  ('李四','男',12000,'2019-03-15', 1, NULL),
  ('王五','女',9500, '2021-06-01', 2, 2000),
  ('赵六','女',6000, '2022-09-20', 2, NULL),
  ('孙七','男',15000,'2018-11-05', 3, 3000),
  ('周八','女',7000, '2023-02-18', NULL, NULL);  -- 周八暂未分配部门，用于演示外连接
```

注意我们特意制造了两条「孤儿」数据：**行政部没有员工**、**周八没有部门**。它们在内连接里会被「过滤掉」，在外连接里又能被「捞出来」，是理解连接差异的关键。

## 一、笛卡尔积：所有错误的起点

如果天真地把两张表写在 `FROM` 后面，不加任何条件：

```sql
SELECT * FROM emp, dept;
```

结果会是 `emp` 的每一行去和 `dept` 的每一行**两两组合**。emp 有 6 行、dept 有 4 行，结果就是 **6 × 4 = 24 行**。这种「行数相乘」的全组合叫**笛卡尔积**。

绝大多数组合是无意义的（比如「张三」配上了「财务部」，但张三明明是研发部）。**笛卡尔积本身没用，多表查询的本质就是：在笛卡尔积的基础上，用条件筛掉无意义的组合，只留下「真正有关联」的行。** 那个条件就是 `emp.dept_id = dept.id`。

> 💡 记住这个思维模型，内连接/外连接都是「笛卡尔积 + 过滤条件」的不同变种。

## 二、内连接 INNER JOIN

> **内连接：只返回两张表中「相互匹配」（连接条件成立）的行。** 没匹配上的（孤儿数据）会被丢弃。

内连接有两种写法，结果完全一样。

### 2.1 隐式内连接（用 WHERE 过滤笛卡尔积）

```sql
-- 查每个员工的姓名、工资 及其所在部门的名称
SELECT emp.ename, emp.salary, dept.dept_name
FROM emp, dept
WHERE emp.dept_id = dept.id;
```

结果（注意：行政部没人、周八没部门，都被过滤掉了）：

| ename | salary | dept_name |
|-------|--------|-----------|
| 张三 | 8000 | 研发部 |
| 李四 | 12000 | 研发部 |
| 王五 | 9500 | 市场部 |
| 赵六 | 6000 | 市场部 |
| 孙七 | 15000 | 财务部 |

### 2.2 显式内连接（INNER JOIN ... ON）

```sql
SELECT emp.ename, emp.salary, dept.dept_name
FROM emp INNER JOIN dept     -- INNER 可省略，写 JOIN 也行
ON emp.dept_id = dept.id;
```

结果与隐式写法**完全相同**。区别只是：连接条件写在 `ON` 里，语义更清晰，`WHERE` 可以专心放业务过滤条件。

> 💡 **推荐用显式 JOIN**：`ON` 管「表怎么连」，`WHERE` 管「连完之后筛哪些行」，职责分明，是 SQL 标准写法。

### 2.3 表别名让 SQL 更简洁

表名太长时用别名（`AS` 可省略）：

```sql
SELECT e.ename, e.salary, d.dept_name
FROM emp e          -- emp 起别名 e
JOIN dept d         -- dept 起别名 d
ON e.dept_id = d.id
WHERE e.salary > 8000;   -- 连接后再加业务条件：只看工资大于 8000 的
```

> ⚠️ **一旦给表起了别名，后面就必须用别名**，不能再用原表名 `emp.ename`，否则报错。

## 三、外连接 OUTER JOIN

内连接会丢弃「孤儿」。但有时我们**就是想看到没匹配上的那一方**——比如「列出所有部门，哪怕这个部门一个人都没有」。这就要用外连接。

### 3.1 左外连接 LEFT JOIN

> **以左表为主**：左表的**所有行都保留**，右表没匹配上的部分用 `NULL` 填充。

```sql
-- 查询所有部门，以及各部门下的员工（没有员工的部门也要列出来）
SELECT d.dept_name, e.ename
FROM dept d                  -- dept 在左，作为主表
LEFT JOIN emp e
ON d.id = e.dept_id;
```

结果（行政部保留了，员工列为 NULL）：

| dept_name | ename |
|-----------|-------|
| 研发部 | 张三 |
| 研发部 | 李四 |
| 市场部 | 王五 |
| 市场部 | 赵六 |
| 财务部 | 孙七 |
| 行政部 | **NULL** |

注意：周八（没部门）**不会出现**，因为左表是 dept，周八不在 dept 里。

### 3.2 右外连接 RIGHT JOIN

> **以右表为主**：右表的所有行都保留，左表没匹配上的用 `NULL` 填充。

```sql
-- 查询所有员工，以及其部门（没有部门的员工也要列出来）
SELECT d.dept_name, e.ename
FROM dept d
RIGHT JOIN emp e            -- emp 在右，作为主表
ON d.id = e.dept_id;
```

结果（周八保留了，部门列为 NULL）：

| dept_name | ename |
|-----------|-------|
| 研发部 | 张三 |
| 研发部 | 李四 |
| 市场部 | 王五 |
| 市场部 | 赵六 |
| 财务部 | 孙七 |
| **NULL** | 周八 |

> 💡 `A LEFT JOIN B` 等价于 `B RIGHT JOIN A`。**实际开发中绝大多数人统一用 `LEFT JOIN`**，把主表放左边，思路更顺。

### 3.3 内连接 vs 外连接 一图看懂

| 连接方式 | 保留谁 | 在我们的数据里 |
|----------|--------|----------------|
| INNER JOIN | 只保留两边都匹配的 | 5 行（不含行政部、不含周八） |
| LEFT JOIN（dept 在左） | 左表(部门)全保留 | 6 行（多了「行政部 / NULL」） |
| RIGHT JOIN（emp 在右） | 右表(员工)全保留 | 6 行（多了「NULL / 周八」） |

> ⚠️ MySQL **不支持** `FULL OUTER JOIN`（全外连接）。需要「两边孤儿都要」时，用 `LEFT JOIN` 的结果 `UNION` 上 `RIGHT JOIN` 的结果来模拟。

## 四、子查询（嵌套查询）

> **子查询：一个 SELECT 语句嵌套在另一个 SQL 语句里**（通常用括号包起来）。内层查询的结果，供外层查询使用。

按**子查询结果的形态**，分三种情况，这是掌握子查询的关键分类。

### 4.1 情况一：结果是「单行单列」——当作一个值，配比较运算符

子查询只返回一个值（一个标量），就可以用 `> < = >= <= != ` 等比较运算符直接拿来比较。

```sql
-- 需求：查询工资高于"全公司平均工资"的员工
-- 第一步：先单独算平均工资
SELECT AVG(salary) FROM emp;     -- 假设结果是 9583.33

-- 第二步：把它塞进 WHERE 当条件值（子查询）
SELECT ename, salary
FROM emp
WHERE salary > (SELECT AVG(salary) FROM emp);
```

结果：李四(12000)、王五(9500 不行，低于9583？看具体值)、孙七(15000)等高于平均的员工。

> 💡 为什么必须用子查询而不能写 `WHERE salary > AVG(salary)`？因为**聚合函数不能直接写在 WHERE 里**（WHERE 在分组聚合之前执行），所以只能先用子查询把平均值算出来。

```sql
-- 类似例子：查工资最高的员工是谁
SELECT ename, salary FROM emp
WHERE salary = (SELECT MAX(salary) FROM emp);
```

### 4.2 情况二：结果是「多行单列」——当作一个集合，配 IN / ANY / ALL

子查询返回一列里的多个值，用 `IN`（在……之中）来匹配。

```sql
-- 需求：查询"研发部"和"市场部"的所有员工
-- 子查询先查出这两个部门的 id（多行单列）
SELECT id FROM dept WHERE dept_name IN ('研发部','市场部');   -- 返回 1,2

-- 外层用 IN 匹配
SELECT ename, salary, dept_id
FROM emp
WHERE dept_id IN (SELECT id FROM dept WHERE dept_name IN ('研发部','市场部'));
```

结果：张三、李四、王五、赵六。

> 💡 `IN (子查询)` 适合「在一组动态值之中」；还有 `> ANY`（大于其中任意一个，即大于最小值）、`> ALL`（大于所有，即大于最大值）等高级用法。

### 4.3 情况三：结果是「多行多列」——当作一张虚拟表，放在 FROM 后

当子查询返回的是多行多列（像一张小表），就可以把它当成一张临时表参与连接，**必须给这张虚拟表起别名**。

```sql
-- 需求：查询 2020-01-01 之后入职的员工 及其部门信息
-- 把"入职晚的员工"先筛成一张虚拟表 t，再和 dept 连接
SELECT t.ename, t.join_date, d.dept_name
FROM (SELECT * FROM emp WHERE join_date > '2020-01-01') t   -- t 是子查询当表，必须起别名
JOIN dept d
ON t.dept_id = d.id;
```

结果：张三、王五、赵六、周八（周八没部门会被内连接过滤）等。

> ⚠️ **FROM 后的子查询（派生表）必须起别名**，否则 MySQL 报 `Every derived table must have its own alias`。

### 4.4 三种子查询用法速查

| 子查询结果形态 | 用在哪 | 配什么 | 例子 |
|----------------|--------|--------|------|
| 单行单列（一个值） | WHERE | `> < = != >= <=` | `salary > (SELECT AVG(salary)...)` |
| 多行单列（一列多值） | WHERE | `IN` / `ANY` / `ALL` | `dept_id IN (SELECT id FROM dept...)` |
| 多行多列（一张表） | FROM | 当虚拟表 JOIN（必须起别名） | `FROM (SELECT...) t JOIN dept d ...` |

## 五、综合练习

> 全部基于上面的 `dept` / `emp` 数据，先自己想，再看答案。

### 练习 1：查询每个员工的姓名、工资、部门名称、部门所在城市

**分析**：员工信息在 emp，部门名和城市在 dept，按 `dept_id = id` 内连接。

```sql
SELECT e.ename, e.salary, d.dept_name, d.loc
FROM emp e
JOIN dept d ON e.dept_id = d.id;
```

### 练习 2：查询每个部门的名称 和 该部门的平均工资（没有员工的部门平均工资显示为 NULL）

**分析**：要保留「没员工的部门」（行政部），所以用 `LEFT JOIN`；再按部门分组求平均。

```sql
SELECT d.dept_name, AVG(e.salary) AS avg_salary
FROM dept d
LEFT JOIN emp e ON d.id = e.dept_id
GROUP BY d.id, d.dept_name;
```

结果里「行政部」的 `avg_salary` 为 `NULL`（它一个员工都没有）。

### 练习 3：查询工资高于其所在部门平均工资的员工

**分析**：典型「相关子查询」——先把「各部门平均工资」算成一张虚拟表，再和员工连接比较。

```sql
SELECT e.ename, e.salary, e.dept_id, t.avg_sal
FROM emp e
JOIN (
    SELECT dept_id, AVG(salary) AS avg_sal
    FROM emp
    GROUP BY dept_id
) t
ON e.dept_id = t.dept_id
WHERE e.salary > t.avg_sal;
```

这道题把**子查询当虚拟表 + 内连接 + 比较**三个知识点全用上了，是多表查询的集大成。

## 六、本章小结

- **笛卡尔积**：两表直接组合产生「行数相乘」的全组合，多数无意义；连接查询 = 笛卡尔积 + 过滤条件。
- **内连接**：只要两边都匹配的行；
  - 隐式：`FROM a, b WHERE a.x = b.y`
  - 显式（推荐）：`FROM a JOIN b ON a.x = b.y`
- **外连接**：保留某一方的全部，缺失方补 NULL；
  - 左外 `LEFT JOIN`（左表为主）、右外 `RIGHT JOIN`（右表为主）；
  - MySQL 无 `FULL JOIN`，用 `UNION` 模拟。
- **子查询三形态**：
  - 单行单列 → WHERE 里配比较运算符；
  - 多行单列 → WHERE 里配 `IN`；
  - 多行多列 → FROM 里当虚拟表（必须起别名）。
- **实战要点**：用表别名、显式 JOIN、ON 管连接 / WHERE 管过滤。

### 常见易错问答

1. **问：内连接和外连接结果不一样，怎么选？**
   答：只要「两边都匹配」的数据用内连接；需要「保留没匹配上的一方」（如没员工的部门）就用外连接。
2. **问：`WHERE` 写连接条件和 `ON` 写有啥区别？**
   答：内连接里两者结果相同；但**外连接里差别很大**——`ON` 里的条件影响「怎么连/补不补 NULL」，`WHERE` 里的条件是连完之后再过滤，可能把补出来的 NULL 行又滤掉，导致外连接「退化」成内连接。
3. **问：子查询和 JOIN 能互换吗？**
   答：很多场景能互换。一般 JOIN 性能更好、可读性更强；子查询在表达「先算个值再比较」时更直观。

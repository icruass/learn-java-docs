# 第 1 章 数据库基本概念与常见数据库软件

> **本章导读**
>

> 读者画像：你已经会写代码（比如 Java），但对数据库不熟。所以本章会**大量用 Java 类比**，帮你把已有的知识迁移过来。

## 1. 为什么需要数据库：先看“不用数据库”有多难受

学任何技术，先问“它解决了什么痛点”，比直接背定义有用得多。

假设你在做一个公司的“员工管理系统”，要存这样的数据：

| 员工编号 | 姓名 | 性别 | 工资  | 入职日期   | 部门  |
| -------- | ---- | ---- | ----- | ---------- | ----- |
| 1        | 张三 | 男   | 8000  | 2020-01-10 | 研发部 |
| 2        | 李四 | 男   | 12000 | 2019-03-15 | 研发部 |
| 3        | 王五 | 女   | 9500  | 2021-06-01 | 市场部 |

> 💡 **提示**：上面这张表，正是我们整套教程会反复使用的 `emp`（员工）表。先混个眼熟，后面每一章都会见到它。

### 1.1 方案 A：用普通文件（txt / CSV / Excel）存

最朴素的想法是：把数据写进一个文件。比如用 CSV：

```text
1,张三,男,8000,2020-01-10,研发部
2,李四,男,12000,2019-03-15,研发部
3,王五,女,9500,2021-06-01,市场部
```

数据量小（几十行）时确实够用。但只要业务一长大，下面这 5 个痛点会接连爆发。

#### 痛点 1：查找慢（没有索引）

需求：“找出工资大于 9000 的员工”。

用文件存，你只能写代码**从头到尾一行行扫描**（全文件遍历）。100 行还好，**100 万行**呢？每查一次都要把整个文件读一遍，慢得让人崩溃。

而数据库可以建**索引**（类比“书的目录 / 字典的偏旁部首检字表”）：不用翻完整本书，直接跳到目标位置。后面专门有一章讲索引，这里先记住一句话：**数据库天生为“高效查找”而生**。

#### 痛点 2：并发访问会出乱子（无并发控制）

需求：A、B 两个人同时给“张三”发工资。

- A 读到张三工资 8000，准备加 1000；
- 与此同时 B 也读到 8000，准备加 2000；
- A 写回 9000，B 写回 10000……

最终张三的工资取决于“谁后写”，其中一笔加薪**凭空消失**了（这叫“丢失更新”）。普通文件没有任何机制阻止这种事，多个程序同时写一个文件，轻则数据错乱，重则文件损坏。

数据库则提供**锁**和**事务**来保证并发安全（这是数据库的核心价值之一，后面有专章讲事务）。

#### 痛点 3：数据冗余 + 不一致（无规范化）

注意上表里“研发部”出现了两次。如果研发部改名叫“技术部”，你得把**每一行**都改一遍。一旦漏改一行，数据库里就同时存在“研发部”和“技术部”，到底哪个对？这就是**冗余导致的不一致**。

数据库可以把“部门”单独拆成一张 `dept` 表，员工表里只存“部门编号”。改名时只改 `dept` 表一处即可——这就是后面要讲的**表的关系/范式设计**。

#### 痛点 4：没有数据校验（无约束）

文件不会拦你：性别字段你写个“abc”、工资写成负数、入职日期写成“昨天”，文件统统照单全收，等到程序读取时才报错（甚至不报错，直接算出离谱结果）。

数据库可以加**约束（Constraint）**：

- 工资必须是数字、不能为负；
- 性别只能是“男/女”；
- 员工编号不能重复（主键）；
- 部门编号必须真实存在（外键）。

不合法的数据，**根本插不进去**——数据从源头就被守住了。

#### 痛点 5：难共享、难备份、难权限管理

文件存在某台电脑上，别的程序、别的同事想用就得拷来拷去；想做“只能看不能改”的权限控制几乎不可能；想定时备份、想多人通过网络同时访问，更是难上加难。

数据库是一个**独立的服务（Server）**，所有人通过网络连接它、按权限访问同一份数据，备份、恢复、审计都有成熟方案。

### 1.2 小结：文件 vs 数据库

| 能力           | 普通文件（txt/Excel）  | 数据库               |
| -------------- | ---------------------- | -------------------- |
| 海量数据查找   | 慢，全文件扫描         | 快，可建索引         |
| 并发读写       | 容易冲突、损坏         | 锁 + 事务保证安全    |
| 数据冗余/一致性| 难控制，易不一致       | 可拆表、加约束       |
| 数据校验       | 无                     | 丰富的约束           |
| 共享/权限/备份 | 困难                   | 天生支持，方案成熟   |

> ✅ 一句话总结：**当数据量大、要被多人/多程序长期、安全、高效地访问时，就该上数据库。**

## 2. DB、DBMS、SQL：三个最容易混的概念

刚入门的人常把这三个词混着说。其实它们分工非常清楚，**一句话区分**：

- **DB（DataBase，数据库）**：存数据的“**仓库**”——是被管理的“数据本身”。
- **DBMS（DataBase Management System，数据库管理系统）**：管理这个仓库的“**软件/管理员**”——比如 MySQL、Oracle 这些软件，本质都是 DBMS。
- **SQL（Structured Query Language，结构化查询语言）**：你和 DBMS 沟通用的“**命令/语言**”——你说“给我查工资大于 9000 的人”，就用 SQL 写出来。

### 2.1 用一个生活类比彻底记住

把整套东西想象成一个**图书馆**：

| 概念  | 类比          | 说明                                             |
| ----- | ------------- | ------------------------------------------------ |
| DB    | 馆里的**书**  | 真正被存储、被查询的数据                         |
| DBMS  | **图书管理员**| 负责存书、找书、保管、不让乱拿的那个“系统”        |
| SQL   | 你说的**话**  | “帮我找本《Java 编程思想》”——你下达指令用的语言 |

你（程序/用户）**从不直接去翻书堆**，而是把需求用 SQL 告诉管理员（DBMS），管理员去仓库（DB）里操作，再把结果递给你。

### 2.2 用 Java 的视角再看一遍

如果你写过 Java，可以这样对应：

- **DBMS** ≈ 你安装在电脑上的那个**软件进程**（比如 `mysqld.exe` 在后台运行），相当于一个一直待命的“服务”。
- **DB** ≈ 这个软件管理的**一组数据文件**（具体存在磁盘上，你不用关心格式）。
- **SQL** ≈ 你的 Java 程序通过 JDBC **发给数据库的那条命令字符串**，例如：

```java
// Java 程序里，最终发给数据库的就是一条 SQL 字符串
String sql = "SELECT * FROM emp WHERE salary > 9000";
```

```sql
-- 这条 SQL 就是“你说的话”，由 DBMS（如 MySQL）负责执行
SELECT * FROM emp WHERE salary > 9000;
```

**它们的关系图（从外到内）：**

```text
用 SQL 下命令
      │
      ▼
┌─────────────────────────────────────┐
│   DBMS（如 MySQL 软件，负责管理）     │
│   ┌─────────────────────────────┐   │
│   │   DB（一个个数据库/仓库）     │   │
│   │   ┌───────────────────────┐ │   │
│   │   │  表 / 行 / 列（数据）  │ │   │
│   │   └───────────────────────┘ │   │
│   └─────────────────────────────┘   │
└─────────────────────────────────────┘
```

> ⚠️ **注意**：日常口语里我们常说“我在用 MySQL 数据库”，这里的“MySQL”其实指的是 **DBMS**（软件），而真正叫“数据库（DB）”的，是 MySQL 里面那一个个你自己建的库（比如我们接下来要建的 `db_learn`）。**软件 ≠ 数据库**，别再混了。

> 🕳️ **常见坑**：很多人以为“SQL 是 MySQL 专属的”。其实 **SQL 是一套国际标准语言**（ISO/ANSI 标准），Oracle、SQL Server、PostgreSQL 都支持。各家会在标准之上加自己的“方言”（比如 MySQL 的 `LIMIT`、SQL Server 的 `TOP`），但**核心语法是通用的**。这意味着你学会 MySQL 的 SQL，换别的数据库也能很快上手。

## 3. 数据库的层级模型：DBMS → 数据库 → 表 → 行/列

这是**全书最重要的一张“心智地图”**，请务必记牢。

### 3.1 四个层级

一个 DBMS（如 MySQL 软件）里，可以装很多个**数据库**；一个数据库里，可以有很多张**表**；一张表，由很多**行**和**列**组成。层层包含，像“俄罗斯套娃”。

```text
DBMS（MySQL 软件）
└── 数据库 db_learn          ← 一个项目通常用一个数据库
    ├── 表 dept（部门表）
    └── 表 emp（员工表）       ← 表是真正存数据的二维结构
        ├── 列：id  ename  gender  salary ...   （字段，定义“有哪些属性”）
        └── 行：(1,张三,男,8000,...)            （记录，一行就是一个具体的员工）
```

各层级的专业术语对照（中英文你都会遇到）：

| 层级 | 中文叫法           | 英文 / 别名                    | 它是什么                       |
| ---- | ------------------ | ------------------------------ | ------------------------------ |
| 1    | 数据库管理系统     | DBMS                           | 管理一切的软件                 |
| 2    | 数据库             | Database / Schema              | 一组相关表的集合（一个项目一库）|
| 3    | 表                 | Table                          | 二维表格，真正存数据的地方     |
| 4    | 行 / 记录          | Row / Record                   | 一行就是一条完整数据           |
| 4    | 列 / 字段          | Column / Field                 | 一列就是一个属性               |

### 3.2 “二维表”长什么样

表（Table）就是我们最熟悉的**二维表格**——有表头（列），有一行行数据（行）。以 `emp` 表为例：

```text
        ┌────┬───────┬────────┬────────┬────────────┬─────────┬───────┐
列名 →  │ id │ ename │ gender │ salary │ join_date  │ dept_id │ bonus │   ← 列(字段)
        ├────┼───────┼────────┼────────┼────────────┼─────────┼───────┤
行(记录)→│ 1  │ 张三  │  男    │  8000  │ 2020-01-10 │    1    │ 1000  │
        │ 2  │ 李四  │  男    │ 12000  │ 2019-03-15 │    1    │ NULL  │
        │ 3  │ 王五  │  女    │  9500  │ 2021-06-01 │    2    │ 2000  │
        └────┴───────┴────────┴────────┴────────────┴─────────┴───────┘
```

- **列（字段）**：定义“一个员工有哪些属性”——编号、姓名、性别、工资……每列还规定了**数据类型**（比如 `salary` 必须是数字）。
- **行（记录）**：每一行是**一个具体的员工**的完整信息。

### 3.3 用 Java 类比：表 = 一个对象的集合

这是给程序员最直观的类比——**一张表，就相当于 Java 里“某个类的对象列表”**：

```java
// 一行(记录) ≈ 一个 Java 对象
class Emp {
    int     id;        // ≈ 列 id
    String  ename;     // ≈ 列 ename
    char    gender;    // ≈ 列 gender
    double  salary;    // ≈ 列 salary
    LocalDate joinDate;// ≈ 列 join_date
    int     deptId;    // ≈ 列 dept_id
    Double  bonus;     // ≈ 列 bonus（用包装类型 Double 才能表示 NULL）
}

// 一张表(table) ≈ 一个 List<对象>
List<Emp> empTable = List.of(
    new Emp(1, "张三", '男', 8000,  ..., 1, 1000.0),
    new Emp(2, "李四", '男', 12000, ..., 1, null),   // bonus 为 null
    new Emp(3, "王五", '女', 9500,  ..., 2, 2000.0)
);
```

对照记忆：

| 数据库概念       | Java 概念              |
| ---------------- | ---------------------- |
| 表（Table）      | 一个 `List<对象>`      |
| 列 / 字段        | 类的一个**成员变量**   |
| 列的数据类型     | 成员变量的**类型**     |
| 行 / 记录        | 一个**对象实例**       |
| `NULL`           | Java 的 `null`         |

> 💡 **提示**：注意上面 `bonus` 用的是 `Double`（包装类）而不是 `double`（基本类型）。因为数据库里 `bonus` 列**允许为 `NULL`**（李四没有奖金），而 Java 基本类型 `double` 无法表示“空”，必须用能装 `null` 的包装类。这正好对应数据库里“**NULL 表示未知/不存在**”的概念——后面讲 `NULL` 时还会专门展开。

> ⚠️ **注意**：上面只是“类比”帮助理解，并不是说表在磁盘上真的是个 List。表底层是 DBMS 用专门的数据结构（如 B+ 树）存储的，但对你写 SQL 来说，**把它当成“带类型约束的二维表 / 对象集合”去理解就完全够用了**。

## 4. 关系型数据库 RDBMS：表 + 关系 + ACID

上面讲的“用二维表存数据、表和表之间有关系”，正是**关系型数据库（RDBMS，Relational DataBase Management System）**的核心思想。我们这套教程的主角 MySQL，就是关系型数据库。

### 4.1 “关系”体现在哪？

“关系型”不只是说“表是一种关系”，更重要的是**表与表之间能建立联系**。回到痛点 3：我们把“部门”从员工表里拆出去，单独建一张 `dept` 表：

```sql
-- 部门表：每个部门一行
CREATE TABLE dept (
    id INT PRIMARY KEY AUTO_INCREMENT,   -- 部门编号
    dept_name VARCHAR(20),               -- 部门名称
    loc VARCHAR(20)                      -- 所在城市
);

INSERT INTO dept (dept_name, loc) VALUES
    ('研发部','北京'),
    ('市场部','上海'),
    ('财务部','广州');
```

`dept` 表的内容：

| id | dept_name | loc  |
| -- | --------- | ---- |
| 1  | 研发部    | 北京 |
| 2  | 市场部    | 上海 |
| 3  | 财务部    | 广州 |

然后员工表 `emp` 只存一个 `dept_id`（部门编号），用它“指向” `dept` 表里的某一行——这就是**外键关系**：

```sql
CREATE TABLE emp (
    id INT PRIMARY KEY AUTO_INCREMENT,   -- 员工编号
    ename VARCHAR(20),                   -- 姓名
    gender CHAR(1),                      -- 性别 男/女
    salary DOUBLE,                       -- 工资
    join_date DATE,                      -- 入职日期
    dept_id INT,                         -- 所属部门(外键 -> dept.id)
    bonus DOUBLE,                        -- 奖金(可能为 NULL)
    CONSTRAINT fk_emp_dept FOREIGN KEY (dept_id) REFERENCES dept(id)
);
```

这样一来：

- 研发部改名，只改 `dept` 表一处，所有员工“自动”跟着变（因为他们只是引用编号）——解决了冗余/不一致。
- `dept_id` 必须是 `dept` 表里**真实存在**的编号，否则插不进去——这就是外键约束的“校验”。

> 💡 **提示**：表间关系有三种典型形态，后面会专章讲，这里先有个印象：
> - **一对多**：一个部门有多个员工（`dept` ↔ `emp`）。
> - **多对多**：一个学生选多门课，一门课有多个学生（用中间表 `student_course` 连接 `student` 和 `course`）。
> - **一对一**：一个人对应一张身份证（`person` ↔ `card`）。

### 4.2 ACID：关系型数据库的“四大金刚”

关系型数据库最被信赖的原因，是它的**事务（Transaction）**满足 **ACID** 四大特性。**事务**就是“一组要么全做、要么全不做的操作”，最经典的例子是**银行转账**：A 给 B 转 100 元 = “A 减 100” + “B 加 100”，这两步必须捆成一个整体。

| 字母 | 特性             | 英文          | 一句话解释                                   | 转账例子                                   |
| ---- | ---------------- | ------------- | -------------------------------------------- | ------------------------------------------ |
| A    | **原子性**       | Atomicity     | 一组操作要么全成功，要么全失败（不可分割）   | “A 扣钱”和“B 加钱”不能只做一半             |
| C    | **一致性**       | Consistency   | 操作前后，数据始终处于“合法/正确”的状态      | 转账前后，两人**总金额不变**               |
| I    | **隔离性**       | Isolation     | 多个事务并发执行时互不干扰                   | 别人同时转账，不会看到你“扣了但没到账”的中间态 |
| D    | **持久性**       | Durability    | 事务一旦提交，结果永久保存（断电也不丢）     | 转账成功后，就算服务器立刻断电，钱也已到账 |

> ⚠️ **注意**：ACID 是关系型数据库相比文件、相比很多 NoSQL 的**核心优势**。涉及钱、订单、库存这类“一分都不能错”的业务，几乎都选关系型数据库。事务和 ACID 后面有专门一整章详讲，这里先记住这四个字母代表什么。

## 5. 非关系型数据库 NoSQL：当“二维表”不够用时

关系型数据库很强，但不是万能的。当数据**结构多变、追求极致性能、数据量爆炸式增长**时，硬塞进二维表反而别扭。于是出现了 **NoSQL**。

### 5.1 NoSQL 到底是什么意思？

`NoSQL` 通常解释为 **“Not Only SQL”（不仅仅是 SQL）**，而不是“拒绝 SQL”。它泛指**一切非关系型的数据库**。它们的共同特点：

- **不强制用二维表**，数据模型灵活；
- **通常不强求 ACID**，更追求高并发、高可扩展（能轻松加机器扩容）；
- 很多没有固定“表结构（schema）”，字段可以随时变。

### 5.2 NoSQL 的四大类

按“数据怎么组织”，NoSQL 主要分四类：

| 类型           | 一句话模型               | 代表产品          | 典型适用场景                                   |
| -------------- | ------------------------ | ----------------- | ---------------------------------------------- |
| **键值型**     | 一个 key 对一个 value，像超大号 `HashMap` | Redis、Memcached  | 缓存、计数器、Session、排行榜——读写极快      |
| **文档型**     | 存“类 JSON”的文档，字段可随意嵌套/变化 | MongoDB           | 结构多变的数据：商品详情、日志、用户配置       |
| **列族型**     | 按“列”存储，适合超大规模稀疏数据 | HBase、Cassandra  | 海量数据分析、时序数据、大数据平台             |
| **图型**       | 用“节点 + 边”存关系       | Neo4j             | 关系网络：社交好友、知识图谱、推荐、风控       |

**直观感受一下“键值型”和“文档型”长啥样：**

```text
键值型 (Redis)：  就像 Java 的 Map<String, String>
   "user:1001:name"  ->  "张三"
   "online_count"    ->  "528"

文档型 (MongoDB)：每条数据是一份独立的 JSON，结构可以各不相同
   {
     "_id": 1, "ename": "张三", "gender": "男",
     "salary": 8000,
     "skills": ["Java", "MySQL"],     // 可以直接存数组
     "address": { "city": "北京" }    // 可以嵌套对象
   }
```

> 💡 **提示**：注意文档型里 `skills` 是个数组、`address` 是个嵌套对象——这在关系型数据库里要拆好几张表才能表达。**“结构灵活”正是 NoSQL 的最大卖点。**

> ⚠️ **注意**：NoSQL **不是用来取代关系型数据库的**，而是“各管一段”。现实项目里常常**两者并用**：核心业务数据（订单、账户）放 MySQL 保证 ACID；缓存、热点数据放 Redis 提速；非结构化日志放 MongoDB。能根据场景选型，才是高手。

## 6. 关系型 vs 非关系型：一张对比表看清楚

| 对比维度       | 关系型数据库（RDBMS）            | 非关系型数据库（NoSQL）              |
| -------------- | -------------------------------- | ------------------------------------ |
| 数据模型       | 二维表（行 + 列），结构固定      | 键值 / 文档 / 列 / 图，结构灵活      |
| 表结构(Schema) | 严格预定义，字段不能乱加         | 大多无固定 schema，字段可随时变      |
| 查询语言       | 标准 SQL                         | 各家自定义 API / 查询方式            |
| 表间关系       | 支持外键、多表 JOIN 关联查询     | 弱，通常不支持 JOIN（需在程序里处理）|
| 事务 / ACID    | 强支持，数据强一致               | 多数较弱（部分支持，常为最终一致）   |
| 扩展方式       | 偏向**纵向扩展**（升级单机配置） | 偏向**横向扩展**（加机器即可）       |
| 性能           | 复杂关联查询强，超高并发吃力     | 简单读写极快，能扛海量高并发         |
| 代表产品       | MySQL、Oracle、PostgreSQL        | Redis、MongoDB、HBase、Neo4j         |
| 典型场景       | 订单、账户、ERP 等强一致业务     | 缓存、日志、海量数据、灵活结构数据   |

> ✅ **选型口诀**：
> - 数据**规整、关系多、要求强一致（钱/订单）** → 关系型（如 MySQL）。
> - 数据**多变、量巨大、要极致读写性能** → NoSQL（如 Redis / MongoDB）。
> - 大型项目往往**两者搭配使用**。

## 7. 常见关系型数据库软件巡礼

市面上的关系型数据库（RDBMS 软件，本质都是 DBMS）很多，了解它们的“定位、收费、特点”有助于你工作中选型，也常出现在面试里。

| 软件                      | 出品方           | 收费情况               | 定位 / 特点                                                  |
| ------------------------- | ---------------- | ---------------------- | ------------------------------------------------------------ |
| **Oracle**                | 甲骨文           | 商业收费（昂贵）       | 功能最强、最稳定的“重量级选手”，大型银行/电信/政府首选，运维复杂、价格高 |
| **MySQL**                 | 现属 Oracle      | 开源免费（有企业版）   | 互联网公司最流行，轻量、快、生态好，**本教程主角**           |
| **Microsoft SQL Server**  | 微软             | 商业收费（有免费版）   | 与 Windows / .NET 生态深度集成，企业内部系统常用             |
| **DB2**                   | IBM              | 商业收费              | IBM 大型机/金融行业的“老牌贵族”，稳定，多见于传统大企业       |
| **PostgreSQL**            | 开源社区         | 开源免费              | 功能强大、标准遵循度高、扩展性强，号称“最先进的开源数据库”，近年极火 |
| **SQLite**                | 开源（公有领域） | 开源免费              | **嵌入式**、零配置、整个库就是一个文件，常用于手机 App、浏览器、小工具 |
| **MariaDB**               | 开源社区         | 开源免费              | MySQL 之父在 Oracle 收购 MySQL 后另起的**分支**，与 MySQL 高度兼容，可平替 |

几点要补充说明：

- **Oracle 系“一家三口”很容易搞混**：甲骨文公司 **Oracle** 同时拥有 **Oracle 数据库**（收费旗舰）和 **MySQL**（开源）。所以“被 Oracle 收购”的是 MySQL 这个产品，而“Oracle 数据库”是另一个东西。
- **SQLite 是个异类**：它**没有独立的服务器进程**，不是“客户端连服务器”的模式，而是直接以一个库文件嵌进你的程序里运行。轻、小、零运维，但不适合高并发、多用户的大型系统。
- **MariaDB 的由来很有故事**：当年 Oracle 收购了 Sun（MySQL 的东家），MySQL 创始人 Monty 担心 MySQL 不再开源，于是带着源码 **fork（分叉）** 出了 MariaDB（名字取自他女儿 Maria），坚持开源。它和 MySQL 几乎可以无缝替换。

> 💡 **提示**：从“学习/找工作”角度看，**MySQL 和 PostgreSQL 是互联网公司用得最多的两个开源关系型数据库**，学会其一，另一个也能很快迁移。

## 8. 常见 NoSQL：Redis 与 MongoDB

NoSQL 产品里，Java 开发者最常打交道的是这两位，单独认识一下。

### 8.1 Redis（键值型，缓存之王）

- **类型**：键值（Key-Value）型，开源免费。
- **最大特点**：数据主要存在**内存**里，所以读写**极快**（每秒可达几万到几十万次操作）。
- **典型用途**：
  - **缓存**：把数据库里的热点数据缓存到 Redis，减轻 MySQL 压力（这是它最经典的角色）；
  - **计数器 / 排行榜**：网站在线人数、点赞数、游戏排行榜；
  - **分布式 Session、分布式锁、消息队列**等。
- **数据结构丰富**：不只是简单的 key-value，还支持 List、Set、Hash、有序集合等，像“加强版的内存数据结构服务器”。

> 💡 **提示**：你几乎可以把 Redis 理解成“一个独立运行、可被所有服务共享、还能持久化到磁盘的超大号 `HashMap`”。

### 8.2 MongoDB（文档型，灵活结构之王）

- **类型**：文档（Document）型，开源（社区版免费）。
- **最大特点**：以**类 JSON 的文档（BSON）**存数据，**无需预先定义表结构**，字段随时增减，嵌套数组/对象随便存。
- **典型用途**：
  - 结构经常变化的数据：商品详情、用户画像、各种配置；
  - 海量日志、爬虫数据、内容管理系统（CMS）；
  - 需要快速迭代、不想频繁改表结构的项目。

> ⚠️ **注意**：Redis 偏“**快**”（缓存、临时高频数据），MongoDB 偏“**灵活 + 海量**”（结构多变的大数据）。两者定位不同，常和 MySQL 一起出现在同一个项目里，各司其职。

## 9. 重头戏：MySQL 的历史与特点（我们为什么选它）

终于轮到主角。理解 MySQL 的来龙去脉，能帮你在版本选择、社区版/企业版等问题上不踩坑。

### 9.1 简要历史脉络

```text
1995 年  瑞典 MySQL AB 公司发布 MySQL，开源、免费、轻量，迅速风靡互联网。
            │
2008 年  Sun Microsystems（升阳）以约 10 亿美元收购 MySQL AB。
            │
2010 年  Oracle（甲骨文）收购 Sun —— MySQL 从此归 Oracle 所有。
            │
            │  （MySQL 创始人 Monty 担忧其开源前景，fork 出了 MariaDB）
            ▼
至今     MySQL 仍是全球最流行的开源关系型数据库之一，被海量互联网公司使用。
```

> 🕳️ **常见坑**：面试常问“MySQL 现在属于哪家公司？”——答案是 **Oracle（甲骨文）**，而不是它最初的 MySQL AB，也不是 Sun。收购链路是：**MySQL AB → Sun → Oracle**。

### 9.2 社区版 vs 企业版

| 版本                        | 是否收费   | 面向人群 / 特点                                       |
| --------------------------- | ---------- | ----------------------------------------------------- |
| **社区版 (Community)**      | 免费、开源 | 个人学习、绝大多数互联网公司用的就是它，功能完全够用   |
| **企业版 (Enterprise)**     | 付费       | 提供额外的监控、备份、安全、技术支持等高级特性与官方服务 |

> 💡 **提示**：我们整套教程、以及你日常学习开发，**用免费的社区版就够了**。下一章安装时，下载的也是社区版。

### 9.3 MySQL 5.x 与 8.x 的主要区别

你在网上会看到 5.7 和 8.0 两个常见版本，简单了解它们的差异（不必死记，有个印象即可）：

| 对比点         | MySQL 5.x（如 5.7）              | MySQL 8.x（如 8.0+）                          |
| -------------- | -------------------------------- | --------------------------------------------- |
| 默认字符集     | `latin1`（易出中文乱码）         | `utf8mb4`（完整支持中文、emoji，**更省心**）  |
| 性能           | 较好                             | 显著优化，整体更快                            |
| 新特性         | 较少                             | 支持**窗口函数**、CTE（`WITH` 公用表表达式）、隐藏索引等现代 SQL 特性 |
| 默认认证插件   | `mysql_native_password`          | `caching_sha2_password`（更安全，但老客户端/驱动可能连不上，需注意） |
| 数据字典       | 部分用表 + 部分用文件            | 统一的事务性数据字典，更可靠                  |

> ⚠️ **注意**：新项目**建议直接用 MySQL 8.x**（默认 `utf8mb4` 省去大量中文乱码烦恼，特性也更全）。
>
> 🕳️ **常见坑**：升到 8.x 后，老的客户端工具或旧版 JDBC 驱动可能因为默认认证插件变成 `caching_sha2_password` 而**连不上数据库**。解决办法是升级驱动，或把用户的认证方式改回 `mysql_native_password`。这个坑后面安装/JDBC 章节会再遇到，先有印象。

### 9.4 MySQL 的核心特点

- **开源免费**：社区版免费，学习、创业零成本。
- **轻量、安装简单、上手快**：相比 Oracle，部署和维护门槛低很多。
- **性能好、够稳定**：完全能扛住绝大多数互联网业务的并发量。
- **跨平台**：Windows、Linux、macOS 都能跑。
- **生态成熟、资料海量**：用的人多，遇到问题几乎都能搜到答案；和 Java、PHP、Python 等语言的整合都非常成熟。
- **支持标准 SQL**：学会它，迁移到别的关系型数据库也很容易。

### 9.5 为什么 Java 学习首选 MySQL？

1. **免费开源**：学习不花钱，环境随便装、随便折腾。
2. **就业刚需**：国内绝大多数 Java 岗位、互联网公司都在用 MySQL，会它=有饭吃。
3. **轻量易学**：对新手友好，不会被复杂的安装和运维劝退，能把精力放在“学 SQL 本身”。
4. **与 Java 生态无缝衔接**：JDBC、MyBatis、Spring Data JPA 等都把 MySQL 当作头号支持对象，学完 SQL 就能顺畅接入 Java 项目（本套教程后面会有 JDBC 实战）。
5. **学一通百**：MySQL 用的是标准 SQL，掌握后再学 Oracle、PostgreSQL 等事半功倍。

> ✅ **结论**：对 Java 开发者来说，**MySQL 是性价比最高的数据库入门选择**——免费、好学、岗位需求大、和 Java 配合好。这也是本套教程选它的原因。

## 本章小结

把本章的核心要点列成清单，回顾一下：

- **为什么要数据库**：相比 txt/Excel 等普通文件，数据库解决了五大痛点——**查找慢、并发乱、数据冗余不一致、无校验、难共享/备份/授权**。
- **DB / DBMS / SQL 三者关系**：
  - DB = 数据本身（仓库里的书）；
  - DBMS = 管理数据的软件（图书管理员，如 MySQL）；
  - SQL = 与 DBMS 沟通的标准语言（你说的话）。
  - 口语里的“MySQL 数据库”通常指 **DBMS 软件**，而真正的“数据库”是它里面的库（如 `db_learn`）。
- **层级模型**：`DBMS → 数据库 → 表 → 行(记录)/列(字段)`。一张表 ≈ Java 里的 `List<对象>`，列 ≈ 成员变量，行 ≈ 对象，`NULL` ≈ `null`。
- **关系型数据库（RDBMS）**：用二维表存数据、表间可建关系（外键、JOIN），事务满足 **ACID（原子性、一致性、隔离性、持久性）**，适合订单/账户等强一致业务。
- **非关系型数据库（NoSQL）**：结构灵活、扩展性强、性能高，分**键值（Redis）、文档（MongoDB）、列族（HBase）、图（Neo4j）** 四类，常作为 RDBMS 的补充与之并用。
- **常见 RDBMS 软件**：Oracle（收费旗舰）、MySQL（开源主流，本教程主角）、SQL Server（微软系）、DB2（IBM 系）、PostgreSQL（强大开源）、SQLite（嵌入式单文件）、MariaDB（MySQL 的开源分支）。
- **MySQL**：1995 诞生，收购链路 **MySQL AB → Sun → Oracle**；分**社区版（免费）/企业版（付费）**；**8.x 默认 `utf8mb4`、性能更好、特性更全**，新项目首选；因免费、易学、岗位需求大、与 Java 生态契合，成为 **Java 学习的首选数据库**。

## 常见面试 / 易错问答

**Q1：DB、DBMS、SQL 有什么区别？**
A：DB 是数据本身（仓库）；DBMS 是管理数据的软件（如 MySQL、Oracle）；SQL 是操作数据库的标准语言。三者是“数据—管理软件—操作语言”的关系。

**Q2：我们平时说“用 MySQL 数据库”，这个说法严谨吗？**
A：不太严谨。MySQL 是一个 **DBMS（软件）**，真正的“数据库（DB）”是 MySQL 内部你创建的那些库（如 `db_learn`）。一个 MySQL 软件里可以装很多个数据库。

**Q3：关系型和非关系型数据库的核心区别？**
A：关系型用结构固定的二维表、支持表间关系和多表 JOIN、事务满足 ACID，适合强一致业务；非关系型结构灵活、易横向扩展、性能高但事务通常较弱，适合缓存、海量、灵活结构的数据。两者常配合使用。

**Q4：ACID 是哪四个？**
A：原子性（Atomicity，全做或全不做）、一致性（Consistency，前后状态合法）、隔离性（Isolation，并发互不干扰）、持久性（Durability，提交后永久保存）。

**Q5：MySQL 现在属于哪家公司？收购历史是怎样的？**
A：现属 **Oracle（甲骨文）**。链路：MySQL AB（瑞典）→ 被 Sun 收购（2008）→ Sun 被 Oracle 收购（2010）。MySQL 创始人后来 fork 出了 MariaDB。

**Q6：MySQL 5.7 和 8.0 最直观的区别是什么？**
A：默认字符集不同——5.x 默认 `latin1`（易中文乱码），8.x 默认 `utf8mb4`（原生支持中文与 emoji）；此外 8.x 性能更好，新增窗口函数、CTE 等特性，默认认证插件也换成了更安全的 `caching_sha2_password`。

**Q7：Redis 和 MySQL 是竞争关系吗？项目里会一起用吗？**
A：不是非此即彼。MySQL（关系型）负责持久、强一致的核心数据；Redis（键值型 NoSQL）做缓存、加速热点读写。实际项目里两者经常**搭配使用**，各司其职。
